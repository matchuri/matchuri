# 열린 의사결정

이 문서는 아직 확정하지 않은 중요한 설계/운영 결정을 관리합니다.
사람이 읽는 의사결정 배경 요약은 GitHub Wiki에서 관리합니다.

## 사용 원칙

- 아직 안 정했지만 작업에 영향을 주는 것만 남깁니다.
- 확정된 내용은 관련 기준 문서로 옮깁니다.
- 임시 방침이 있으면 함께 적고, 최종 결정 전까지는 그 범위 안에서만 작업합니다.

## Open

### Redis 도입 시점

- 현재 방침: 즉시 도입하지 않습니다.
- 결정 시점: 그룹 추천/투표의 실시간 요구가 커지거나 성능 이슈가 발생할 때
- 판단 기준: 세션 공유, 캐싱, SSE fan-out, 로그아웃 전략 중 실제 병목이 확인되는가

### 비회원 추천 범위

- 현재 방침: 비회원 추천 기능은 존재하지만 회원 전용 기능과 동일한 권한 경계로 취급하지 않습니다.
- 결정 시점: 개인 추천 API와 프론트 온보딩 흐름 연결 전
- 판단 기준: 일회성 개인 추천이 가입 전환과 제품 이해에 도움이 되는가

### Frontend 기술 스택 최종 확정

- 현재 방침: 현재 코드베이스 기준을 유지하되, UI/상태관리/데이터 패칭 패턴은 별도 확정합니다.
- 결정 시점: 프론트 구현 규칙을 본격 문서화할 때

### Member 도메인의 Custom Repository 도입 시점

- 현재 방침: 기본 `JpaRepository`를 우선 사용합니다.
- 결정 시점: 서비스에 쿼리 조립 책임이 생기거나 동적 조회 요구가 늘어날 때

## Closed 요약

- Refresh Token 전략: 로그인 단위 다중 Refresh Token 저장
- 로그아웃 의미: 현재 로그인 세션의 Refresh Token 삭제와 cookie 제거
- Infisical 시크릿 주입 방식: GitHub Actions OIDC 기반 주입
- 소셜 로그인 도입 시점: Google OAuth2를 MVP 인증 범위에 포함
- 그룹 투표 상태 반영 방식: 1차 구현은 SSE
- 자체 로그인 이메일 필수화와 계정 복구 정책: 신규 자체 회원가입부터 이메일 인증 필수

## 닫는 방법

- 결정이 끝나면 상태를 `Closed`로 바꿉니다.
- 최종 결론을 한 줄로 적습니다.
- 관련 API, 데이터, 백엔드, 프론트엔드 기준 문서에 반영합니다.
