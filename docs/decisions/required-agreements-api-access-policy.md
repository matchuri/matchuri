# 필수 약관 미동의 회원 API 접근 정책

이 문서는 Matchuri에서 필수 약관 동의를 완료하지 않은 회원이 어떤 API를 사용할 수 있고, 어떤 API는 차단되는지 구현 전에 정리한 설계 문서입니다.
목적은 백엔드 인가 기준과 프론트 라우팅 흐름이 서로 어긋나지 않도록 경계를 먼저 맞추는 것입니다.

## 문서 목적

- 로그인 가능과 핵심 서비스 이용 가능을 구분하는 기준을 명확히 합니다.
- 미동의 회원이 허용되는 API와 차단되는 API를 분리합니다.
- Security 구현 전에 어떤 경로를 약관 동의 완료 조건으로 묶을지 합의합니다.

## 현재 확정 전제

- 미동의 회원도 로그인은 가능합니다.
- 필수 약관 동의 완료 여부는 `member_agreements` 기준으로 판단합니다.
- 필수 약관은 `TERMS_OF_SERVICE`, `PRIVACY_POLICY` 두 종류입니다.
- 프론트는 로그인 직후 약관 동의 완료 여부를 확인하고 미완료면 약관 동의 화면으로 이동합니다.

## 왜 별도 접근 정책 문서가 필요한가

- 인증은 되었지만 서비스 이용 조건은 미완료인 상태를 표현해야 합니다.
- 약관 상태 조회와 동의 제출 API까지 막아버리면 사용자가 완료 상태로 이동할 수 없습니다.
- 반대로 핵심 API를 너무 넓게 열어두면 약관 동의 강제 흐름이 무너집니다.

즉, Matchuri에서는 "공개 API / 로그인 필요 API / 필수 약관 완료 필요 API"를 구분해야 합니다.

## 접근 레벨 정의

### 1. 공개 API

- 로그인하지 않아도 호출할 수 있는 API

예:

- 회원 가입
- 로그인
- 로그인 ID 중복 확인
- OAuth2 시작 및 콜백

### 2. 로그인만 필요 API

- 로그인은 필요하지만, 필수 약관 동의 완료 전에도 호출할 수 있는 API

역할:

- 사용자가 약관 동의를 완료 상태로 이동할 수 있게 돕는 API

### 3. 필수 약관 완료 필요 API

- 로그인과 필수 약관 동의 완료가 모두 필요한 API

역할:

- 실제 Matchuri 핵심 기능 수행

## 권장 정책 표

| 구분 | 예시 API | 미동의 회원 허용 여부 | 이유 |
| --- | --- | --- | --- |
| 공개 API | `POST /api/v1/members` | 허용 | 계정 생성 자체는 가능해야 함 |
| 공개 API | `POST /api/v1/auth/login` | 허용 | 로그인 자체는 허용 정책 |
| 공개 API | `GET /api/v1/auth/oauth2/{provider}` | 허용 | OAuth2 로그인 시작점 |
| 공개 API | `GET /login/oauth2/code/{provider}` | 허용 | OAuth2 콜백 처리 |
| 로그인만 필요 API | `POST /api/v1/auth/logout` | 허용 | 미동의 상태에서도 세션 종료 가능해야 함 |
| 로그인만 필요 API | `GET /api/v1/member-agreements/required-status` | 허용 | 약관 상태 확인을 위해 필요 |
| 로그인만 필요 API | `POST /api/v1/member-agreements/consents` | 허용 | 약관 동의 완료를 위해 필요 |
| 로그인만 필요 API | `POST /api/v1/auth/oauth2/exchange` | 허용 | 소셜 로그인 후 후속 토큰 교환 필요 |
| 필수 약관 완료 필요 API | `GET /api/v1/members/me` 주요 기능 | 차단 | 서비스 진입 조건 통제 |
| 필수 약관 완료 필요 API | `PATCH /api/v1/members/me` | 차단 | 기본 프로필 수정도 서비스 사용 흐름 일부 |
| 필수 약관 완료 필요 API | `PATCH /api/v1/members/me/taste-profile` | 차단 | 온보딩 다음 단계이므로 약관 완료 후 허용 |
| 필수 약관 완료 필요 API | 개인 추천 API 전반 | 차단 | 핵심 서비스 기능 |
| 필수 약관 완료 필요 API | 그룹 생성/참여/탈퇴 | 차단 | 핵심 서비스 기능 |
| 필수 약관 완료 필요 API | 그룹 추천/투표/확정 | 차단 | 핵심 서비스 기능 |

가정(Assumption):

- `GET /api/v1/members/me`는 최소 프로필 확인 용도로 일부 허용하는 선택지도 있을 수 있습니다.
- 현재는 "핵심 서비스 진입 전 혼선을 줄인다"는 쪽으로 더 보수적으로 잡아, 주요 보호 API로 분류합니다.

## 권장 인가 흐름

### 1. 로그인 전

- 공개 API만 허용

### 2. 로그인 후, 필수 약관 미완료

- 로그인만 필요 API 허용
- 필수 약관 완료 필요 API 차단

### 3. 로그인 후, 필수 약관 완료

- 핵심 보호 API 허용

## 프론트와의 연결 기준

프론트는 로그인 성공 후 아래 순서를 따르는 것이 권장됩니다.

1. 필수 약관 상태 조회
2. 미완료면 약관 동의 화면으로 이동
3. 동의 제출 성공 후 홈 또는 다음 온보딩 단계로 이동

이때 백엔드가 핵심 API를 차단해주면, 프론트 라우팅 누락이 있더라도 서비스 진입 조건이 무너지지 않습니다.

즉, 프론트 강제 이동은 UX 장치이고, 백엔드 차단은 정책 보장 장치입니다.

## 왜 `members/me`를 보수적으로 보는가

`members/me`는 겉보기에는 단순 조회처럼 보이지만, 실제로는 로그인 이후 서비스 상태 확인과 프로필 진입의 시작점이 될 수 있습니다.

현재 Matchuri에서는 아래 이유로 보수적으로 잡는 편이 적절합니다.

- 약관 동의 전 상태에서 서비스 내부 화면으로 자연스럽게 흘러가는 것을 줄이고 싶습니다.
- 첫 흐름은 "약관 동의 완료"에 집중돼야 합니다.
- 2인 팀 운영 기준에서는 예외를 많이 두기보다 명확한 경계가 유지보수에 유리합니다.

다만 구현 시점에 "최소 프로필 확인은 허용"이 더 낫다고 판단되면 재검토할 수 있습니다.

## 실패 응답 기준

- 미동의 회원이 필수 약관 완료 필요 API를 호출하면 `403`을 반환합니다.
- 권장 에러 코드는 `MEMBER_AGREEMENT_REQUIRED`입니다.

이 응답은 인증 실패가 아니라, 로그인은 되었지만 현재 이용 조건을 충족하지 못한 상태를 의미합니다.

## Security 구현 시 권장 방향

- 공개 경로는 기존 화이트리스트 방식 유지
- 로그인만 필요 API와 필수 약관 완료 필요 API를 별도 묶음으로 관리
- `hasCompletedRequiredAgreements(memberId)` 같은 기준 메서드로 공통 판단

가정(Assumption):

- 최초 구현은 커스텀 필터, 인가 서비스, 또는 메서드 레벨 체크 중 가장 단순한 방식으로 시작할 수 있습니다.
- 현재 단계에서는 구현 기법보다 정책 경계 확정이 더 중요합니다.

## 검증 시나리오 초안

- 미동의 회원은 약관 상태 조회 API를 호출할 수 있다.
- 미동의 회원은 약관 동의 제출 API를 호출할 수 있다.
- 미동의 회원은 로그아웃할 수 있다.
- 미동의 회원은 개인 추천 API를 호출할 수 없다.
- 미동의 회원은 그룹 관련 API를 호출할 수 없다.
- 약관 동의 완료 후에는 핵심 보호 API에 접근할 수 있다.

## 현재 권장 결론

- Matchuri는 필수 약관 미동의 회원에 대해 "로그인은 가능하지만 핵심 서비스는 차단" 정책을 유지한다.
- 이를 위해 API를 `공개 API`, `로그인만 필요 API`, `필수 약관 완료 필요 API`로 구분한다.
- 약관 상태 조회와 동의 제출 API는 반드시 열어두고, 추천/그룹/프로필 수정 같은 핵심 기능은 필수 약관 완료 후에만 허용하는 것이 가장 적절하다.
