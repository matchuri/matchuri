# 데이터 문서 작성 가이드

이 문서는 Matchuri에서 `docs/data/` 아래 데이터 문서를 어떤 방식으로 작성하고 유지할지 정리합니다.
목적은 엔티티와 스키마가 늘어나도 문서 구조와 톤이 흔들리지 않게 만드는 것입니다.

## 왜 이 가이드가 필요한가

- 현재 구현 기준 데이터는 점점 늘어나고 있습니다.
- 회원/인증처럼 테이블별로 쪼개는 편이 좋은 영역이 있고, 메뉴 카탈로그처럼 묶어서 설명하는 편이 좋은 영역도 있습니다.
- 2인 팀 운영에서는 문서 스타일이 매번 바뀌기보다, "어디에 어떤 깊이로 쓸지"가 일정한 편이 유지보수에 유리합니다.

## 기본 원칙

- 현재 기준 문서는 `docs/`에 둡니다.
- 데이터 문서의 공식 진입점은 `docs/data/index.md`입니다.
- 현재 구현된 테이블 문서의 진입점은 `docs/data/implemented-jpa-data-model.md`입니다.
- 구현과 문서가 다르면 문서보다 구현을 먼저 확인하고, 이후 `docs/`를 갱신합니다.
- 오래 유지될 구조 판단은 실행 계획 문서가 아니라 `docs/data/` 또는 관련 설계 문서로 승격합니다.

## 어떤 문서를 써야 하는가

### 1. 현재 구현 테이블 문서

대상:

- 이미 JPA 엔티티나 DDL로 구현된 테이블

위치:

- `docs/data/`

예:

- `members-schema.md`
- `member-agreements-schema.md`
- `menu-catalog-schema.md`

목적:

- "지금 코드 기준으로 테이블이 어떻게 생겼는가"를 설명

### 2. 구현 전 상세 설계 문서

대상:

- 아직 구현 전이지만 구조, 제약, 버전 관리 기준을 먼저 합의해야 하는 테이블 또는 데이터 구조

위치:

- `docs/data/`

예:

- `member-required-agreements-schema.md`

목적:

- 구현 전에 팀 기준을 맞추는 설계 초안

주의:

- 구현이 완료되면 구현 기준 문서와의 관계를 명확히 적습니다.
- 구현 전 초안이 현재 기준처럼 읽히지 않게 "초안", "구현 전" 성격을 문서 안에 드러냅니다.

## 문서를 개별 파일로 쪼갤지, 묶음 문서로 둘지

### 개별 파일로 분리하는 경우

- 테이블 하나만 읽어도 이해가 필요한 경우
- 테이블별 제약, 상태값, 운영 규칙 차이가 큰 경우
- 회원, 인증, 이력처럼 책임이 분명한 경우
- 이후 문서 분량이 더 커질 가능성이 높은 경우

예:

- `members`
- `member_agreements`
- `auth_refresh_tokens`

### 묶음 문서로 두는 경우

- 서로 강하게 연결된 기준 데이터 집합인 경우
- 한 테이블만 떼어 설명하면 오히려 이해가 떨어지는 경우
- 메뉴 마스터 + 매핑 테이블처럼 함께 읽어야 맥락이 사는 경우

예:

- `attribute_categories`
- `ingredients`
- `menu_items`
- `menu_attribute_categories`
- `menu_ingredients`

현재 기준:

- 회원/인증 영역은 개별 문서
- 메뉴 카탈로그 영역은 묶음 문서 허용

## 권장 문서 구조

현재 구현 테이블 문서는 아래 흐름을 기본으로 합니다.

1. 문서 목적
2. 현재 확정 전제
3. 왜 이 구조를 택했는가
4. 테이블 개요
5. 현재 컬럼 정의
6. 컬럼 해석
7. 현재 제약조건
8. 현재 명시 보조 인덱스
9. enum 기준 또는 상태값 기준
10. 연관관계
11. 현재 쿼리 패턴 초안
12. 현재 판단
13. 함께 볼 문서

모든 문서가 위 항목을 전부 가져야 하는 것은 아닙니다.
다만 최소한 아래는 가능한 한 유지합니다.

- 문서 목적
- 현재 확정 전제
- 테이블 개요
- 현재 컬럼 정의
- 제약조건 또는 명시 보조 인덱스 정책
- 현재 판단
- 함께 볼 문서

## 문체와 톤

- 설명형 설계 문서 톤을 기본으로 합니다.
- 단순 테이블 사전처럼 컬럼만 나열하지 않습니다.
- "왜 이 구조가 필요한가", "운영상 어떤 판단이 들어갔는가"를 함께 적습니다.
- 미확정 내용은 `가정(Assumption):`으로 분리합니다.
- 구현 전 초안과 구현 기준 문서를 혼동하지 않도록 표현을 분명히 나눕니다.

## 표 사용 기준

컬럼 정의는 가능하면 표로 정리합니다.

권장 컬럼:

- 컬럼명
- 타입
- NULL
- 제약/기타
- 설명

표만으로 끝내지 말고, 해석이 필요한 컬럼은 별도 섹션에서 풀어 씁니다.

예:

- `code`
- `is_active`
- `sort_order`
- 상태 enum 컬럼

## 링크 기준

- 연관 테이블 문서가 있으면 본문 안에 링크를 넣습니다.
- 데이터 문서 인덱스와 상호 이동이 가능해야 합니다.
- `함께 볼 문서` 섹션에는 관련 문서를 2~5개 정도 연결합니다.

최소 링크 권장:

- `docs/data/index.md`
- `docs/data/implemented-jpa-data-model.md`
- 직접 참조하는 연관 테이블 문서
- 필요 시 `docs/data/implemented-jpa-data-model.md`

## 구현과 문서의 관계

### 구현이 먼저인 경우

- 엔티티, DDL, 마이그레이션을 먼저 확인합니다.
- 현재 구현 기준으로 문서를 씁니다.
- 구현과 과거 문서가 다르면 과거 문서를 그대로 답습하지 않습니다.

### 설계가 먼저인 경우

- 구현 전 상세 설계 문서로 초안을 씁니다.
- 구현이 완료되면 현재 구현 기준 문서 또는 묶음 문서에 반영합니다.

## 스키마 변경 시 체크리스트

- 엔티티 또는 DDL 변경
- 필요 시 마이그레이션 스크립트 추가
- `docs/data/implemented-jpa-data-model.md` 갱신
- 관련 `docs/data/*.md` 갱신
- `docs/data/index.md` 또는 `implemented-jpa-data-model.md` 링크 갱신
- 용어가 바뀌면 `docs/decisions/domain-language.md` 확인

## 현재 권장 운영 방식

- 구현된 테이블은 `docs/data/`에 현재 기준 문서로 설명합니다.
- 문서 수가 많아져도 `index.md`와 `implemented-jpa-data-model.md`를 통해 진입점을 유지합니다.
- 메뉴 카탈로그처럼 결합이 강한 영역은 묶음 문서를 허용합니다.
- 회원/인증처럼 책임이 뚜렷한 영역은 개별 문서를 우선합니다.
