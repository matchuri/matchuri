# 백엔드 배포 인프라 결정

이 문서는 Matchuri 백엔드 MVP 단계에서 어떤 배포 인프라 구조를 채택하고, 왜 그렇게 결정하는지 정리합니다.
핵심 기준은 확장성 과시가 아니라 `BE 1 + FE 1` 팀이 실제로 운영 가능한 단순함입니다.

## 문서 목적

- 백엔드 배포 구조의 source of truth를 정합니다.
- 실제 인프라 구축 전에 운영 복잡도와 책임 경계를 먼저 맞춥니다.
- 프론트엔드 완성 전에도 백엔드 API를 독립 배포해 검증할 수 있는 기준을 만듭니다.

## 현재 결정

- 백엔드는 AWS EC2 단일 서버에 배포합니다.
- 애플리케이션은 Spring Boot JAR로 실행합니다.
- 데이터베이스는 같은 EC2 내부의 MySQL로 운영합니다.
- 배포 자동화는 GitHub Actions를 사용합니다.
- Frontend는 Vercel에 별도 배포합니다.

## 왜 지금 배포 인프라를 먼저 정하는가

- 프론트엔드가 아직 완성되지 않았더라도, 백엔드 API는 독립적으로 배포해 검증할 수 있어야 합니다.
- 인증, 추천, 그룹 세션처럼 상태가 있는 기능은 로컬 개발만으로 운영 리스크를 충분히 확인하기 어렵습니다.
- 소수 팀에서는 구현 후에 인프라를 한꺼번에 붙이는 방식이 오히려 문맥 전환 비용을 키우기 쉽습니다.
- 따라서 MVP 중간 시점에 "운영 가능한 최소 배포 구조"를 먼저 고정하는 편이 전체 흐름을 더 단순하게 만듭니다.

## 결정 원칙

- 가장 적은 구성 요소로 API를 외부에 안정적으로 노출할 수 있어야 합니다.
- 배포 실패 시 사람이 이해 가능한 절차로 복구할 수 있어야 합니다.
- 백엔드 담당자가 인프라와 배포를 함께 맡는 현실을 기준으로 선택합니다.
- 지금 당장 필요 없는 관리형 서비스나 분산 구성을 서두르지 않습니다.

## 목표 아키텍처

### 구성 요소

- `GitHub`: 소스 저장소와 배포 트리거 관리
- `GitHub Actions`: 백엔드 빌드, 테스트, 배포 자동화
- `EC2`: 백엔드 애플리케이션과 MySQL이 함께 동작하는 단일 서버
- `Nginx`: 외부 요청 수신과 Spring Boot 애플리케이션 프록시
- `Spring Boot JAR`: 실제 API 런타임
- `MySQL`: MVP 서비스 데이터 저장
- `Infisical`: 시크릿 원본 저장소

### 요청 흐름

1. 사용자가 Vercel에 배포된 프론트엔드에 접속합니다.
2. 프론트엔드는 백엔드 공개 도메인으로 API를 호출합니다.
3. EC2의 `Nginx`가 외부 요청을 받아 Spring Boot 애플리케이션으로 전달합니다.
4. Spring Boot 애플리케이션은 동일 서버 내부 MySQL에 접근합니다.

## 운영 방식 기본안

- 가정(Assumption): EC2 운영체제는 Ubuntu LTS를 사용합니다.
- 가정(Assumption): 애플리케이션 프로세스는 `systemd`로 관리합니다.
- 가정(Assumption): 외부 80/443 포트는 `Nginx`가 점유하고, Spring Boot는 내부 포트에서만 실행합니다.
- 가정(Assumption): HTTPS는 `Nginx`와 인증서 자동 갱신 도구 조합으로 처리합니다.

이 기본안은 현재 팀이 인터넷 자료를 가장 쉽게 참고할 수 있고, 문제 발생 시 로그 위치와 서비스 상태 확인 방법도 단순하다는 장점이 있습니다.

## 구성 요소별 책임

### GitHub Actions

- `main`, `develop` 기준 CI를 수행합니다.
- 배포 브랜치 또는 수동 트리거 기준으로 JAR 배포를 수행합니다.
- 서버에 직접 빌드 책임을 넘기지 않고, 검증된 산출물을 전달하는 방향을 우선합니다.
- 시크릿은 GitHub 저장소 비밀값에 장기 보관하지 않고, Infisical OIDC 인증으로 실행 시점에 가져옵니다.

### EC2

- 백엔드 런타임과 MySQL을 함께 운영합니다.
- 단일 장애 지점이지만, 초기 운영 복잡도를 낮추는 대신 의도적으로 선택합니다.
- 운영 로그, 서비스 상태, 디스크 용량을 한 서버에서 추적할 수 있습니다.

### Nginx

- 외부 진입점 역할을 맡습니다.
- HTTPS 종료 지점으로 사용합니다.
- 향후 정적 에러 페이지, 기본 접근 제어, 요청 제한 같은 최소 운영 기능을 붙이기 쉽습니다.

### systemd

- 서버 재시작 후 애플리케이션 자동 기동을 담당합니다.
- 기동 실패, 재시작 이력, 표준 로그 확인이 쉽습니다.
- 별도 프로세스 매니저를 새로 학습하지 않아도 됩니다.

### MySQL

- 회원, 취향, 추천, 그룹 세션, 투표 등 핵심 서비스 데이터를 저장합니다.
- EC2 내부 운영으로 시작하므로 백업과 디스크 관리가 중요합니다.

## 왜 Docker/ECS를 지금 쓰지 않는가

- 지금 단계에서는 컨테이너 오케스트레이션보다 서비스 흐름 완성과 운영 절차 명확화가 더 중요합니다.
- Docker와 ECS는 일관성 장점이 있지만, 팀 규모 대비 학습과 장애 대응 복잡도를 함께 늘립니다.
- 현재 구조에서는 JAR + systemd만으로도 배포 자동화와 재기동 운영이 가능합니다.

## 왜 RDS를 지금 쓰지 않는가

- EC2 내부 MySQL은 고가용성과 백업 편의성에서는 불리하지만, MVP 초기 비용과 구조 복잡도를 낮춥니다.
- 현재 우선순위는 데이터 모델과 API 흐름을 빠르게 검증하는 것입니다.
- DB 운영 책임을 팀이 직접 감수하는 대신, 구조를 더 쉽게 이해하고 통제할 수 있습니다.

## 왜 Redis를 지금 쓰지 않는가

- 현재 MVP 핵심은 "후보 3개 + 투표 + 최종 선택" 흐름 완결입니다.
- 실시간 동기화 고도화보다 상태 흐름 명확성이 먼저입니다.
- 캐시, 세션 저장, Pub/Sub가 꼭 필요한 병목이 아직 확인되지 않았습니다.

## 배포 환경 구분

- `local`: 개발자 로컬 실행
- `test`: 자동 테스트 실행
- `dev`: 공유 서버 또는 staging 성격의 배포 환경
- `prod`: production 배포 환경
- 가정(Assumption): 초기에는 `dev` 성격의 단일 공유 서버를 먼저 운영합니다.

현재 단계에서는 환경 수를 과하게 늘리지 않고, "공유 서버 1대에서 프론트와 연동 가능한 API를 검증한다"는 목적을 우선합니다.

## 장점

- 구성이 적어 팀이 전체 구조를 빠르게 이해할 수 있습니다.
- 문제 발생 지점을 좁히기 쉽습니다.
- 백엔드와 DB를 같은 서버에서 운영하므로 초기 네트워크 구성 부담이 낮습니다.
- GitHub Actions로 반복 가능한 배포 절차를 만들기 쉽습니다.

## 한계와 인정하는 트레이드오프

- EC2와 MySQL이 같은 서버에 있으므로 단일 장애 지점입니다.
- 무중단 배포와 수평 확장을 기본 전제로 하지 않습니다.
- DB 백업과 서버 자원 관리 책임이 팀에 직접 있습니다.
- 서버 자원 부족 시 API와 DB가 함께 영향을 받을 수 있습니다.

이 한계는 문서에서 숨길 대상이 아니라, 현재 팀이 단순함을 얻기 위해 의식적으로 감수하는 비용입니다.

## 지금 채택하는 것

- EC2 단일 서버
- EC2 내부 MySQL
- GitHub Actions 기반 자동화
- Spring Boot JAR 배포
- Nginx + systemd 조합
- Infisical 기반 시크릿 원본 관리
- GitHub Actions OIDC 기반 Infisical 시크릿 주입

## 지금 채택하지 않는 것

- Docker 기반 배포
- ECS/EKS
- RDS
- Redis
- 다중 인스턴스
- 블루그린/카나리 같은 고급 배포 전략

## 운영 성공 기준

- 프론트엔드가 호출 가능한 공개 백엔드 엔드포인트가 있다.
- 배포 절차가 문서와 자동화 기준으로 반복 가능하다.
- 환경 변수와 시크릿 주입 기준이 정리되어 있다.
- 장애 시 1차 확인 순서와 복구 기준이 있다.

## 재검토 시점

아래 조건이 분명해지면 현재 결정을 다시 검토합니다.

- 서버 자원 사용량이 지속적으로 높아진다.
- MySQL 백업/복구 부담이 운영 리스크로 커진다.
- 배포 중단 시간이 팀에 실제 문제를 일으킨다.
- 실시간 그룹 상태 동기화 요구가 커져 Redis 필요성이 생긴다.
- 운영 환경이 한 대 서버로 감당하기 어려운 수준으로 늘어난다.
