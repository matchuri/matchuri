# 필수 약관 동의 구조 결정

## 문서 목적

- 회원 가입 이후 필수 약관 동의를 어떤 구조로 처리할지 구현 전에 기준을 정한다.
- 자체 로그인과 소셜 로그인 신규 가입을 같은 정책으로 다루기 위한 도메인 모델을 정리한다.
- 단순 체크박스 UI 추가가 아니라, 서버 기준에서 "서비스 이용 가능 상태"를 어떻게 판단할지 합의한다.

## 현재 문제 인식

현재 Matchuri 회원 가입 구조는 매우 얇다.

- 자체 회원 가입은 `loginId + password` 중심으로 동작한다.
- 소셜 로그인 신규 사용자도 로그인 성공 시 회원을 즉시 생성하는 방향으로 정리되어 있다.
- 현재 `members` 모델과 스키마 요약에는 약관 동의 이력 또는 버전 개념이 없다.

이 상태에서 회원 가입에 `약관 동의`가 추가되면 아래 문제가 생긴다.

- 프론트 체크박스만으로는 서버가 필수 동의 완료 여부를 보장할 수 없다.
- 자체 로그인과 소셜 로그인 신규 가입이 서로 다른 기준으로 흘러갈 수 있다.
- 약관 개정 시 누가 어떤 버전에 동의했는지 판별할 수 없다.
- 계정 생성과 서비스 이용 가능 상태를 같은 것으로 취급하면 가입 이후 동의 흐름을 표현하기 어렵다.

## 이번에 확정한 조건

- 현재는 필수 약관 동의만 고려한다.
- 필수 약관은 `서비스 이용약관`, `개인정보 처리방침` 두 종류다.
- 소셜 로그인 신규 가입에도 동일한 동의 기준을 적용한다.
- 최신 필수 버전은 서버 상수로 관리한다.
- 미동의 회원은 로그인은 가능하지만, 핵심 API는 차단한다.
- 프론트는 로그인 직후 약관 동의 화면으로 강제 이동한다.

## 판단 원칙

- Matchuri는 2인 팀 운영이므로 법적/운영 요구를 만족하면서도 구조가 과하게 무거워지지 않아야 한다.
- 회원 생성과 약관 동의 완료를 분리하되, 서비스 접근 제어 기준은 서버가 일관되게 판단해야 한다.
- 현재 MVP에서는 선택 약관, 마케팅 수신 동의, 복잡한 정책 엔진까지 도입하지 않는다.
- 향후 약관 버전 변경과 재동의 요구를 감당할 수 있어야 한다.
- 자체 로그인과 소셜 로그인의 가입 후 흐름이 서로 어긋나지 않아야 한다.

## 고려한 옵션

### 옵션 1. 프론트 체크박스만 추가

- 장점:
  가장 빠르게 적용할 수 있다.
- 단점:
  서버가 필수 동의 여부를 보장하지 못한다.
  우회가 가능하고, 동의 버전과 시각이 남지 않는다.
- 판단:
  Matchuri 기준에서는 임시방편에 가깝고 장기 운영 기준으로 적절하지 않다.

### 옵션 2. `members`에 동의 상태 필드 추가

- 장점:
  구현 범위가 상대적으로 작다.
  가입 API에서 최소한의 서버 검증은 가능하다.
- 단점:
  약관 종류와 버전이 늘어날수록 `members`가 비대해진다.
  어떤 버전에 동의했는지 남기기 어렵고, 이력 관리가 부자연스럽다.
- 판단:
  단기 MVP 최소선으로는 가능하지만, 약관 개정 대응을 생각하면 재작업 가능성이 높다.

### 옵션 3. 별도 `member_agreements` 테이블 도입

- 장점:
  회원별 약관 종류, 버전, 동의 시각을 자연스럽게 저장할 수 있다.
  자체 로그인과 소셜 로그인에 같은 구조를 적용하기 쉽다.
  이후 선택 약관이나 재동의 정책이 생겨도 확장 가능하다.
- 단점:
  엔티티, API, 접근 제어, 문서 영향 범위가 더 넓다.
- 판단:
  현재 요구와 이후 확장 가능성을 함께 고려했을 때 가장 균형이 좋다.

## 최종 결정

- 필수 약관 동의는 별도 `member_agreements` 테이블로 관리한다.
- `members`에는 약관 동의 여부를 직접 누적하지 않는다.
- 서비스 이용 가능 여부는 "회원이 현재 필수 약관 2종의 최신 필수 버전에 모두 동의했는가"로 판단한다.

즉, Matchuri에서 `계정 생성`과 `핵심 기능 이용 가능 상태`는 같은 개념이 아니다.

## 목표 구조

### 한 줄 요약

- `Member`는 계정 자체를 나타낸다.
- `member_agreements`는 약관 동의 이력과 현재 충족 상태를 판단하는 근거 데이터를 나타낸다.

### 권장 테이블

- `member_agreements`

권장 컬럼 초안:

- `id`
- `member_id`
- `agreement_type`
- `agreement_version`
- `agreed_at`
- `created_at`
- `updated_at`

권장 제약:

- `(member_id, agreement_type, agreement_version)` unique

권장 `agreement_type` 값:

- `TERMS_OF_SERVICE`
- `PRIVACY_POLICY`

가정(Assumption):

- MVP에서는 "동의 철회"보다 "필수 동의 완료 여부와 버전 기록"이 더 중요하므로, 최초 구조에는 철회 컬럼을 필수로 두지 않는다.

## 최신 필수 버전 관리

최신 필수 버전은 별도 마스터 테이블이 아니라 서버 상수로 관리한다.

예시 방향:

- `TERMS_OF_SERVICE_REQUIRED_VERSION`
- `PRIVACY_POLICY_REQUIRED_VERSION`

이 결정의 이유:

- 현재 필수 약관 종류가 2개로 고정되어 있다.
- 운영팀 규모가 작으므로 초기에는 DB 기반 관리 화면이나 마스터 테이블보다 코드 상수가 더 단순하다.
- 약관 개정이 생겨도 우선은 서버 배포 기준으로 함께 반영 가능하다.

현재는 "어떤 버전이 최신 필수 버전인가"를 서버 코드가 authoritative 하게 판단한다.

## 서버 판단 기준

서버는 아래 질문으로 회원의 약관 상태를 판단한다.

- 이 회원은 `TERMS_OF_SERVICE` 최신 필수 버전에 동의했는가
- 이 회원은 `PRIVACY_POLICY` 최신 필수 버전에 동의했는가

두 조건을 모두 만족할 때만 필수 약관 동의 완료 상태로 본다.

권장 서비스 메서드 예시:

- `hasCompletedRequiredAgreements(memberId)`

이 메서드는 단순 조회 유틸이 아니라, 이후 핵심 API 접근 제어의 기준이 된다.

## 상태 흐름

### 1. 계정 생성

- 자체 회원 가입 또는 소셜 로그인 신규 가입으로 `Member`가 생성된다.
- 이 시점에는 아직 필수 약관 동의가 완료되지 않았을 수 있다.

### 2. 로그인 성공

- 회원은 로그인할 수 있다.
- 다만 서버는 필수 약관 동의 완료 여부를 함께 판단한다.

### 3. 약관 동의 필요 상태 진입

- 필수 약관 2종 중 하나라도 최신 버전 동의가 없으면 `agreement required` 상태로 본다.
- 프론트는 로그인 직후 전용 약관 동의 화면으로 이동한다.

### 4. 약관 동의 제출

- 프론트는 `서비스 이용약관`, `개인정보 처리방침` 각각의 최신 필수 버전에 대한 동의 정보를 서버에 제출한다.
- 서버는 이를 `member_agreements`에 저장한다.

### 5. 핵심 기능 이용 가능 상태 진입

- 서버가 두 필수 약관 모두 최신 필수 버전에 동의한 것을 확인하면 이후 핵심 API 접근을 허용한다.

## 인증/인가 방향

이번 결정에서 중요한 점은 `미동의 회원도 로그인은 가능하다`는 것이다.

즉, Matchuri는 아래 두 단계를 구분한다.

- 인증 성공
- 핵심 서비스 이용 가능

권장 방향:

- 로그인, 로그아웃, 약관 상태 조회, 약관 동의 제출은 허용한다.
- 취향 프로필 관리, 개인 추천, 그룹 생성/참여, 그룹 추천, 투표 같은 핵심 API는 필수 약관 동의 완료 후에만 허용한다.

이 방식의 장점:

- 자체 로그인과 소셜 로그인 신규 가입 후 흐름을 동일하게 가져갈 수 있다.
- 프론트는 로그인 직후 일관된 리다이렉트 규칙을 적용할 수 있다.
- "아직 계정은 있지만 서비스 이용 조건은 미완료"인 상태를 자연스럽게 표현할 수 있다.

## API 방향 초안

구현 전 기준으로 최소 아래 성격의 API가 필요하다.

### 1. 필수 약관 상태 조회

- 현재 로그인한 회원이 필수 약관 동의를 완료했는지 조회한다.
- 부족한 약관 종류를 함께 반환할 수 있다.

예시 역할:

- `requiredAgreementsCompleted`
- `missingAgreementTypes`

### 2. 필수 약관 동의 제출

- 현재 로그인한 회원이 필수 약관 2종에 동의한 사실을 서버에 저장한다.
- 요청에는 `agreementType`, `agreementVersion`이 포함된다.
- 서버는 요청 버전이 현재 최신 필수 버전과 일치하는지 검증한다.

가정(Assumption):

- 동일 회원이 같은 약관 타입과 버전에 대해 중복 제출하더라도, MVP에서는 idempotent 하게 처리하는 편이 운영상 유리하다.

## 프론트 UX 방향

프론트는 로그인 직후 아래 규칙을 따른다.

1. 로그인 성공
2. 필수 약관 완료 여부 확인
3. 미완료면 약관 동의 화면으로 강제 이동
4. 동의 완료 후에만 홈 또는 다음 온보딩 단계로 이동

이 방향은 Matchuri의 제품 원칙과도 맞는다.

- 첫 흐름이 헷갈리지 않아야 한다.
- 사용자가 지금 무엇을 해야 하는지 명확해야 한다.
- 서비스 이용 조건을 흐린 상태로 방치하지 않는다.

## 이 구조를 선택한 이유

- 약관 버전 변경에 대응할 수 있다.
- 소셜 로그인 신규 가입과 자체 회원 가입을 같은 기준으로 처리할 수 있다.
- `members` 엔티티에 약관 전용 상태를 계속 덧붙이지 않아도 된다.
- 현재 팀 규모에서도 감당 가능한 수준의 복잡도다.

즉, `member_agreements`는 지금 당장 조금 더 무겁지만, 이후 재작업 비용을 크게 줄여준다.

## 지금 당장 하지 않는 것

- 선택 약관 관리
- 마케팅 수신 동의
- 약관 전문 자체를 DB에서 관리하는 CMS 구조
- 관리자 약관 버전 관리 화면
- 철회/재동의 이력의 복잡한 상태 머신

현재 MVP에서는 필수 약관 2종의 최신 필수 버전 동의 여부를 안정적으로 판단하는 것이 우선이다.

## 현재 권장 결론

- Matchuri는 `계정 생성 후 약관 동의` 흐름을 지원해야 하므로, 서버는 약관 동의 완료 여부를 독립적으로 판단할 수 있어야 한다.
- 이를 위해 `member_agreements` 테이블을 도입하고, 필수 약관 2종의 최신 필수 버전 동의 여부를 핵심 API 접근 조건으로 사용한다.
- 로그인은 허용하되, 미동의 회원은 전용 약관 동의 화면으로 유도하고 핵심 서비스 API는 차단한다.
- 이 방향이 현재 팀 규모, 제품 흐름, 소셜 로그인 확장성을 함께 고려했을 때 가장 적절하다.
