# 백엔드 코드 품질 기준

이 문서는 Matchuri 백엔드 코드의 품질 기준을 정의합니다.
목적은 "좋은 코드가 무엇인지"를 팀 기준으로 맞추고, 구현과 리뷰에서 같은 판단 언어를 쓰도록 하는 것입니다.

이 문서는 점수를 매기기 위한 표가 아닙니다.
품질을 수치로 평가할 때는 `docs/backend/quality-score.md`를 사용하고, 그때의 판단 근거로 이 문서를 참고합니다.

## 이 문서와 다른 문서의 역할

- `docs/backend/guide.md`
  - 패키지 구조, 구현 위치, 실무 작업 규칙을 다룹니다.
- `docs/backend/quality-score.md`
  - 현재 변경이나 코드베이스 상태를 항목별로 평가하고 우선순위를 정하는 점수표입니다.
- `docs/decisions/backend-code-quality.md`
  - 좋은 백엔드 코드의 설계/구현/리뷰 기준 자체를 정의합니다.

즉, 이 문서는 "어떻게 작성해야 하는가", `QUALITY_SCORE`는 "현재 어느 수준인가"를 다룹니다.

## 범위

- `backend/` 아래 Spring Boot 백엔드 코드
- 주 대상: `api`, `domain`, `global`, `infra` 계층
- 포함: 객체지향적 설계, 테스트 용이성, 변경 용이성, 예외 처리, 의존성 관리, 리뷰 기준
- 제외: 배포 절차, 인프라 운영, 문서 점수화, 프론트엔드 코드 스타일

## 기본 관점

- Matchuri 백엔드는 소수 팀이 장기간 운영할 수 있어야 합니다.
- "구조적으로 멋져 보이는 코드"보다 "이해하기 쉽고 수정하기 쉬운 코드"를 우선합니다.
- 객체지향 원칙은 추상화 자체가 목적이 아니라 책임 분리와 변경 비용 절감을 위해 사용합니다.
- 테스트하기 어려운 코드는 대개 설계도 좋지 않을 가능성이 높다고 봅니다.
- 중복 제거는 중요하지만, 성급한 공통화로 의도를 숨기지 않습니다.

## 좋은 백엔드 코드의 기준

### 1. 책임이 분명하다

- 클래스와 메서드는 한 가지 핵심 책임에 집중합니다.
- Controller는 HTTP 요청/응답과 입력 검증 경계를 다룹니다.
- Service는 유스케이스와 도메인 규칙을 다룹니다.
- Repository는 조회/저장 책임에 집중합니다.
- Mapper는 변환 전용으로 유지하고, 비즈니스 판단을 넣지 않습니다.

좋은 예:

- Controller가 DTO를 받고 Service에 유스케이스를 위임한다.
- Service가 상태 전이와 권한 확인을 처리한다.

나쁜 신호:

- Controller에 분기와 상태 판단이 많다.
- Mapper가 조회, 조건 판단, 예외 발생까지 담당한다.
- Repository가 유스케이스 흐름을 사실상 결정한다.

### 2. 의존 방향이 단순하다

- 상위 흐름은 `api -> domain -> repository/infra` 방향을 기본으로 합니다.
- 하위 계층이 상위 계층 DTO나 웹 개념을 알지 않도록 합니다.
- 도메인 로직이 프레임워크 세부사항에 과도하게 묶이지 않도록 합니다.

좋은 예:

- Service가 API 응답 DTO 대신 도메인 또는 내부 결과 모델을 다룬다.
- 공통 예외와 에러 코드는 계층을 가로질러 일관되게 사용한다.

나쁜 신호:

- 도메인 계층이 `HttpServletRequest`, 응답 DTO, Swagger 어노테이션에 의존한다.
- 여러 계층이 서로의 내부 구현을 직접 안다.

### 3. 추상화는 필요할 때만 도입한다

- 인터페이스, 전략 객체, Custom Repository, 공통 유틸은 반복과 변동 지점이 분명할 때 도입합니다.
- "나중에 필요할 것 같아서" 넣는 추상화는 기본적으로 보수적으로 봅니다.
- 동일한 로직이 여러 곳에서 반복되기 전에는 구현을 가까이 두는 편이 낫습니다.

좋은 예:

- 외부 연동 구현이 여러 개 생기기 시작할 때 인터페이스를 도입한다.
- 복잡한 동적 조회가 생겼을 때만 Custom Repository를 분리한다.

나쁜 신호:

- 구현체가 하나뿐인 인터페이스가 전 계층에 기본처럼 깔린다.
- 작은 중복을 없애려다 공통 메서드가 오히려 읽기 어렵다.

### 4. 상태와 규칙이 코드에서 읽힌다

- 도메인 규칙은 분산된 if문보다, 읽을 수 있는 메서드/구조로 표현합니다.
- 상태 전이는 가능한 한 한 곳에서 관리합니다.
- 예외는 규칙 위반을 드러내는 방식으로 사용하고, 흐름 제어용으로 남용하지 않습니다.

좋은 예:

- `withdrawMember()`, `validateActiveMember()`처럼 의도가 드러나는 메서드가 있다.
- 상태 전이 조건이 Service 또는 도메인 메서드에서 일관되게 관리된다.

나쁜 신호:

- 같은 상태 검사가 여러 API와 Service에 복붙된다.
- 왜 실패하는지 메시지나 에러 코드만 봐서는 규칙을 알기 어렵다.

### 5. 테스트하기 쉬운 구조다

- 비즈니스 로직과 프레임워크/외부 의존성을 가능한 한 분리합니다.
- 테스트를 위해 객체를 만들거나 상태를 세팅하는 비용이 과하지 않아야 합니다.
- 한 테스트에서 너무 많은 mock을 요구하는 구조는 품질 경고로 봅니다.
- 순수 계산/판단 로직은 가능한 한 독립적으로 검증할 수 있어야 합니다.

좋은 예:

- 핵심 규칙은 단위 테스트나 슬라이스 테스트로 빠르게 검증할 수 있다.
- 외부 시스템, 시간, 인증 정보 같은 가변 의존성은 주입 가능하거나 경계에 모여 있다.

나쁜 신호:

- 서비스 하나를 테스트하려면 웹 계층, DB, 보안 컨텍스트를 모두 띄워야 한다.
- 테스트보다 fixture 준비 코드가 더 길다.
- mock이 과도하게 많아 테스트가 구현 세부사항에 묶인다.

### 6. 변경 비용이 낮다

- 작은 요구사항 변경이 여러 파일로 불필요하게 번지지 않아야 합니다.
- 메서드와 클래스 이름만 봐도 수정 지점을 예측할 수 있어야 합니다.
- 중복은 단순 복붙 자체보다 "같은 규칙이 여러 군데에서 따로 진화하는 상태"를 더 위험하게 봅니다.

좋은 예:

- 필드 검증 정책 변경 시 영향 범위가 명확하다.
- 도메인 규칙 수정이 Controller, DTO, Repository까지 줄줄이 퍼지지 않는다.

나쁜 신호:

- 사소한 요구사항 수정에도 에러 코드, 분기, 매핑, 예외 처리 로직이 여러 군데에서 따로 바뀐다.
- 한 메서드가 조회, 검증, 상태 변경, 응답 조립, 로깅을 모두 처리한다.

### 7. 예외와 로그가 운영 친화적이다

- 예외는 의도와 원인을 추적 가능하게 표현합니다.
- 로그는 운영에 필요한 정보만 남기고 민감 정보는 피합니다.
- 에러 응답은 일관된 형식과 코드 체계를 유지합니다.

좋은 예:

- 예상 가능한 실패는 일관된 에러 코드로 표현된다.
- 운영 중 문제를 추적할 수 있는 최소 맥락이 로그에 남는다.

나쁜 신호:

- `RuntimeException`을 임의로 던지고 상위에서 뭉뚱그린다.
- 비밀번호, 토큰, 민감한 요청값이 로그에 남는다.

## 리뷰 시 체크할 질문

코드 리뷰에서는 아래 질문을 우선 봅니다.

### 설계 품질

- 이 클래스는 한 가지 이유로만 변경되는가
- 계층 책임이 섞이지 않았는가
- 이 추상화는 지금 정말 필요한가
- 의존 방향이 자연스러운가

### 테스트 용이성

- 핵심 규칙을 빠르게 검증할 수 있는가
- 이 구조는 mock 없이도 테스트 가능한 부분이 충분한가
- 테스트가 구현 세부사항보다 동작을 검증하는가

### 가독성

- 메서드 이름만 보고 의도를 이해할 수 있는가
- 조건 분기와 상태 전이가 한 번에 읽히는가
- 중복 제거가 오히려 이해 비용을 높이지 않았는가

### 운영 친화성

- 실패 시 어떤 로그와 에러 코드가 남는지 분명한가
- 설정값과 민감 정보가 적절한 경계에 있는가
- 운영자가 문제를 좁혀갈 단서가 남는가

## 강한 경고 신호

아래 상황은 리팩토링 또는 설계 재검토 후보로 봅니다.

- Controller가 사실상 비즈니스 로직을 수행한다.
- Service 메서드 하나가 지나치게 길고 여러 역할을 가진다.
- 같은 규칙 검증이 여러 계층에 중복된다.
- DTO, Entity, 응답 조립이 강하게 엉켜 있다.
- 테스트 하나를 위해 과도한 mock이나 환경 구성이 필요하다.
- 구현체가 하나뿐인 인터페이스가 습관적으로 추가된다.
- 예외와 에러 코드가 계층마다 제각각이다.

## 허용하는 절충

Matchuri는 2인 팀 운영을 전제로 하므로 아래 절충은 허용합니다.

- 작은 중복은 성급한 공통화보다 낫습니다.
- 단일 API에만 필요한 검증은 구현 가까이에 둘 수 있습니다.
- 완벽한 DDD 스타일보다 현재 유스케이스를 읽기 쉬운 구조를 우선합니다.
- 초기 단계에서는 "테스트 가능한 단순 구조"가 "추상화가 많은 정교한 구조"보다 낫습니다.

## QUALITY_SCORE와 연결하는 방법

`docs/backend/quality-score.md`로 품질을 평가할 때 이 문서는 아래 항목의 판단 근거로 씁니다.

- 구조 단순성
- 변경 용이성
- 테스트/검증 가능성
- 운영 준비도

단, 점수와 임계치는 `QUALITY_SCORE`에서 관리합니다.
이 문서에는 점수표, 퍼센트, 등급 기준을 중복해서 적지 않습니다.
