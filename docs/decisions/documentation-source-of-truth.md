# 문서 기준 위치

이 문서는 Matchuri 저장소에서 "어떤 문서를 어떤 목적으로 볼지"를 정리합니다.
목적은 `docs/`와 GitHub Wiki가 함께 존재해도 팀과 에이전트가 같은 기준 위치를 바라보게 만드는 것입니다.

## 기본 원칙

- `docs/`는 개발 기준 문서입니다.
- GitHub Wiki는 사람이 읽는 프로젝트 위키입니다.
- 구현과 함께 계속 갱신해야 하는 문서는 `docs/` 또는 코드 인접 위치에 둡니다.
- 프로젝트 소개, 포트폴리오 서사, 협업 방식, 주요 결정의 읽기 쉬운 요약은 GitHub Wiki에 둡니다.
- 운영 런북, 실행 계획, 내부 회고, 데이터 수집 스크립트, 과거 원문 아카이브는 공개 문서 표면에서 제외합니다.
- 공개 가능한 문서만 보고도 현재 제품 방향, API, 데이터, 백엔드 구조의 큰 흐름을 이해할 수 있어야 합니다.

## 문서 공간 역할

| 위치 | 역할 | 기준성 |
| --- | --- | --- |
| `README.md` | 외부 방문자용 첫 화면과 주요 링크 | 요약 |
| GitHub Wiki | 사람이 읽는 프로젝트 소개, 포트폴리오, 협업 안내 | 요약과 해설 |
| `AGENTS.md` | 에이전트가 항상 따라야 하는 짧은 라우터와 검증 명령 | 실행 기준 |
| `.agents/skills/` | 반복되는 에이전트 작업 절차와 선택적 reference/script | 재사용 워크플로 |
| harness scripts/tests | 기계적으로 검증 가능한 문서/계약 규칙 | 자동 검증 |
| `docs/` | 구현과 함께 갱신하는 개발 기준 문서 | 현재 기준 |
| 코드와 마이그레이션 | 실제 동작과 계약의 최종 확인 위치 | 최종 기준 |
| `secrets/` | 운영 런북, 실행 계획, 내부 기록 | 내부 기준 |

## ERD 관리 원칙

### 현재 기준 위치

- 현재 구현 테이블 인덱스: `docs/data/implemented-jpa-data-model.md`
- 컬럼 단위 설명: `docs/data/*-schema.md`
- 구조, 용어, 설계 판단 기준: `docs/`
- 사람용 요약: 필요 시 GitHub Wiki
- 실제 엔티티/DDL/JPA 매핑: 코드와 마이그레이션, 엔티티 구현

### 업데이트 규칙

- 스키마 구조 판단이 바뀌면 구현과 `docs/data/` 문서를 함께 갱신합니다.
- 문서와 구현이 다르면 엔티티, init SQL, 마이그레이션 기준으로 확인을 시작합니다.

## API 문서 관리 원칙

### 현재 기준 위치

- 계약 기준: 코드와 함께 유지되는 OpenAPI 메타데이터와 그 산출물
- 문서화 전략: `docs/decisions/api-docs-strategy.md`
- API 상태표와 도메인별 설명: `docs/api/`

### 업데이트 규칙

- API 계약이 바뀌면 OpenAPI, 관련 `docs/api/` 설명 문서, 필요한 테스트를 함께 수정합니다.
- 공개 문서에 남기는 API 설명은 현재 구현과 클라이언트 계약을 이해하는 데 필요한 수준으로 유지합니다.

## 문서 작성 판단표

### `docs/`에 둘 것

- 오래 유지할 개발 기준 중 공개 가능한 것
- 팀 공통 규칙
- 현재 구조와 책임 분리
- 현재 데이터 모델 설명
- 현재 API 문서화 전략
- 제품/도메인 기준 용어

### GitHub Wiki에 둘 것

- 프로젝트 소개와 문제 정의
- 포트폴리오용 기능 설명
- 핵심 사용자 흐름과 화면 설명
- 아키텍처와 기술 선택의 읽기 쉬운 요약
- 협업 방식과 버전 관리 안내
- 트러블슈팅, 회고, 학습 내용 중 공개 가능한 요약

### `.agents/skills/`에 둘 것

- 반복되는 에이전트 작업 절차
- API 계약 변경, 데이터 모델 변경, 문서 슬림화처럼 여러 단계가 필요한 workflow
- 필요할 때만 읽을 reference와 실행 가능한 helper script
- `AGENTS.md`에 두기에는 길고, `docs/`에 두기에는 절차적인 내용

### harness로 둘 것

- 금지 링크 검사
- OpenAPI와 API 상태표 drift 검사
- 엔티티/마이그레이션과 데이터 문서 drift 검사
- 문서 크기, 중복 제목, 필수 인덱스 누락처럼 자동 판정 가능한 규칙

### 공개 문서에서 제외할 것

- 운영 런북과 인프라 세부값
- 실행 계획과 작업 이력
- 내부 회고와 장애 원문 기록
- 데이터 수집/가공 스크립트와 원천 데이터
- 과거 원문 아카이브
- secret, 내부 workspace ID, 계정/인프라 식별자

## 실무 체크리스트

- 새 팀원이 "최신 개발 기준이 어디 있지?"라고 물으면 `docs/`로 안내할 수 있어야 합니다.
- 외부 방문자가 "무슨 프로젝트지?"라고 물으면 `README.md`와 GitHub Wiki로 안내할 수 있어야 합니다.
- 반복 절차가 길어지면 `docs/`가 아니라 `.agents/skills/`로 분리되어야 합니다.
- 사람이 눈으로 기억해야 하는 문서 규칙은 가능한 한 harness로 검증되어야 합니다.
- 공개 가능한 문서만 읽어도 현재 구현 방향을 따라갈 수 있어야 합니다.
- 내부 문서나 과거 아카이브가 없어도 공개 문서의 핵심 흐름이 끊기지 않아야 합니다.
- README에는 프로젝트 요약과 문서 진입점을 명확히 적습니다.

## 현재 결론

- ERD와 스키마 설명은 `docs/data/`를 현재 기준으로 둡니다.
- API는 코드 인접 OpenAPI와 `docs/api/`를 현재 계약 기준으로 둡니다.
- 제품 소개, 포트폴리오 서사, 협업 안내는 GitHub Wiki를 사람용 위키로 둡니다.
- 반복되는 에이전트 작업 절차는 `.agents/skills/`로 둡니다.
- 검사 가능한 문서 규칙은 harness script나 테스트로 둡니다.
- 내부 문서와 과거 자료는 공개 기준 문서가 아니라 보조 자료로만 취급합니다.
