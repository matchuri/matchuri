# 품질 점수표

이 문서는 Matchuri의 품질을 "막연한 감"이 아니라, 반복 측정 가능한 기준으로 관리하기 위한 점수표입니다.
점수표의 목적은 평가 자체가 아니라 다음 작업 우선순위를 정하는 데 있습니다.

특히 Matchuri는 추천 정확도만 높은 서비스가 아니라, 개인과 그룹이 점심 메뉴를 빠르게 합의하도록 돕는 서비스이므로 품질 평가도 제품 흐름, 코드 구조, 테스트, 운영 가능성을 함께 봅니다.

## 이 문서의 사용 목적

- 기능 구현 후 현재 품질 수준을 빠르게 진단합니다.
- 낮은 점수 항목을 기준으로 리팩토링 또는 문서 보강 대상을 정합니다.
- PR 리뷰, 회고, 실행 계획 문서에서 공통 언어로 사용합니다.
- 소수 팀이 유지 가능한 수준의 품질을 꾸준히 올립니다.

## 평가 원칙

- 품질은 통과/실패가 아니라 상대적 점수로 관리합니다.
- 점수는 코드 인상비평이 아니라 확인 가능한 근거로 매깁니다.
- 가능하면 사람 판단만 쓰지 않고 자동화 지표를 함께 사용합니다.
- 점수는 구현 복잡도보다 사용자 가치와 운영 단순성을 우선 반영합니다.
- 점수가 낮더라도 당장 제품 가치를 막지 않으면 메모만 남기고 넘길 수 있습니다.
- 미확정 판단은 단정하지 말고 `가정(Assumption):`으로 적습니다.

## 자동화 지표 사용 원칙

주관적 판단을 완전히 없앨 수는 없지만, 아래처럼 자동화 가능한 증거가 있으면 우선 사용합니다.

- CI 결과: 빌드, 테스트, 린트 통과 여부
- 테스트 커버리지: 라인, 브랜치, 핵심 패키지 기준 커버리지
- 정적 분석: 린트, 타입 체크, 포맷, 의존성 검사
- 변경 규모: 파일 수, diff 크기, 큰 파일 또는 긴 함수 증가 여부
- 운영 안전성: 필수 설정 누락 탐지, 보안 검사, 배포 파이프라인 성공 여부

자동화 지표는 "점수를 대신"하기보다 "점수 논쟁을 줄이는 보조 근거"로 사용합니다.
아직 도구가 없는 항목은 억지로 숫자를 만들지 않고, `도입 전` 상태로 기록합니다.

## 현재 측정 가능한 자동화 신호

현재 저장소에서 바로 확인 가능한 자동화 신호는 아래와 같습니다.

| 영역 | 현재 상태 | 비고 |
| --- | --- | --- |
| Backend 테스트 실행 | 가능 | `backend/build.gradle.kts`에서 `useJUnitPlatform()` 기반 테스트 실행이 설정되어 있습니다. |
| Backend CI | 있음 | `backend/.github/workflows/backend-ci.yml`에서 `./gradlew clean build`를 실행합니다. |
| Backend 테스트 커버리지 수집 | 가능 | JaCoCo 리포트를 `backend/build/reports/jacoco/test/` 아래에 생성할 수 있습니다. |
| Frontend 린트 | 가능 | `frontend/package.json`의 `npm run lint`가 존재합니다. |
| Frontend 테스트 커버리지 수집 | 없음 | Frontend에는 아직 커버리지 도구 설정이 없습니다. |
| Frontend 자동 테스트 | 없음 | 현재 `test` 스크립트나 테스트 러너 설정이 없습니다. |

## 점수 스케일

모든 항목은 `0-5점`으로 채점합니다.

| 점수 | 의미 | 해석 기준 |
| --- | --- | --- |
| 0 | 없음 | 기준, 구현, 문서, 검증 흔적이 거의 없습니다. |
| 1 | 매우 부족 | 최소 형태는 있으나 실제 사용이나 유지보수에 바로 쓰기 어렵습니다. |
| 2 | 기초 수준 | 동작은 보이지만 누락, 모호함, 수동 의존이 많습니다. |
| 3 | 실무 가능 | 현재 팀 규모에서 운영 가능한 기본 수준입니다. |
| 4 | 좋음 | 구조, 검증, 문서가 비교적 잘 맞물리고 반복 작업에 강합니다. |
| 5 | 매우 좋음 | 현재 단계에서 기대하는 수준을 충분히 넘었고, 회귀 위험도 낮습니다. |

## 평가 항목

기본 평가는 아래 7개 항목으로 합니다.
항목 수를 불필요하게 늘리기보다, 실제 리팩토링 판단에 직접 연결되는 것만 유지합니다.

| 항목 | 질문 | 대표 근거 |
| --- | --- | --- |
| 제품 적합성 | 이 변경이 Matchuri의 핵심 문제인 "빠른 메뉴 합의"에 기여하는가 | 제품 명세, UX 흐름, 상태 전이, 설명 가능성 |
| 도메인/용어 일관성 | 도메인 경계와 용어가 문서/코드/API에서 일치하는가 | 엔티티/DTO/문서 명칭, 상태명, 에러 코드 |
| 구조 단순성 | 2인 팀이 이해하고 운영하기 쉬운 구조인가 | 책임 분리, 의존 방향, 과도한 추상화 여부 |
| 변경 용이성 | 이후 기능 추가나 수정 시 안전하게 바꾸기 쉬운가 | 응집도, 중복, 결합도, 파일 크기, 함수 복잡도 |
| 테스트/검증 가능성 | 동작을 빠르게 확인하고 회귀를 잡을 수 있는가 | 테스트, 린트, 수동 검증 절차, 실패 재현 가능성 |
| 문서/계약 명확성 | 구현 의도와 외부 계약이 팀에 명확히 전달되는가 | OpenAPI, 설계 문서, 실행 계획, 코드 주석 |
| 운영 준비도 | 배포 후 문제를 파악하고 복구하기 쉬운가 | 로그, 예외 처리, 설정 분리, 신뢰성/보안 기준 부합 여부 |

## 항목별 상세 채점 가이드

### 1. 제품 적합성

- 0점: 기능이 제품 목표와 거의 연결되지 않거나 오히려 흐름을 방해합니다.
- 1점: 기능은 있으나 개인/그룹 의사결정 맥락이 반영되지 않습니다.
- 2점: 핵심 흐름과 연결되지만 설명 가능성, 합의 UX, 상태 정의가 부족합니다.
- 3점: 현재 제품 명세에 맞고 핵심 사용자 흐름을 크게 해치지 않습니다.
- 4점: 사용자 결정 비용 감소에 직접 기여하고, 이유와 상태가 비교적 명확합니다.
- 5점: 개인 추천과 그룹 합의 흐름 모두에서 높은 일관성과 설득력을 가집니다.

### 2. 도메인/용어 일관성

- 0점: 용어가 제각각이라 코드와 문서를 연결하기 어렵습니다.
- 1점: 핵심 용어 일부만 맞고 나머지는 혼용됩니다.
- 2점: 주요 용어는 맞지만 DTO, API, 문서 중 일부가 어긋납니다.
- 3점: 현재 기준 용어가 대부분 맞고, 큰 혼동은 없습니다.
- 4점: 상태, 엔티티, API 계약, 문서가 안정적으로 정렬됩니다.
- 5점: 신규 참여자도 용어만으로 구조를 따라가기 쉬울 정도로 명확합니다.

### 3. 구조 단순성

- 0점: 책임이 섞여 있어 변경 영향 범위를 예측하기 어렵습니다.
- 1점: 동작은 하지만 과한 추상화나 중복 계층이 많습니다.
- 2점: 구조는 보이나 예외 처리나 책임 경계가 아직 흐립니다.
- 3점: 현재 팀이 유지 가능한 수준으로 역할이 분리되어 있습니다.
- 4점: 단순하면서도 확장 포인트가 명확합니다.
- 5점: 운영 복잡도를 거의 늘리지 않으면서 기능 확장까지 잘 수용합니다.

### 4. 변경 용이성

- 0점: 작은 수정도 넓은 영역을 함께 건드려야 합니다.
- 1점: 중복과 결합이 많아 수정이 자주 깨집니다.
- 2점: 주요 흐름은 수정 가능하지만 일부 구간이 병목입니다.
- 3점: 일반적인 기능 추가와 리팩토링이 무리 없이 가능합니다.
- 4점: 변경 지점이 명확하고 회귀 위험이 비교적 낮습니다.
  자동화 근거 예시: 변경 파일 수와 영향 범위가 제한적이고, 관련 테스트 또는 린트가 함께 통과합니다.
- 5점: 의도 파악, 수정, 검증까지의 비용이 매우 낮습니다.
  자동화 근거 예시: 핵심 모듈에 대한 테스트와 커버리지가 안정적으로 유지되고, 작은 변경이 큰 diff로 번지지 않습니다.

### 5. 테스트/검증 가능성

- 0점: 검증 방법이 없거나 사람 기억에 의존합니다.
- 1점: 일부 수동 확인만 가능하고 회귀 방지가 거의 없습니다.
- 2점: 기본 테스트나 린트는 있으나 핵심 흐름 검증이 부족합니다.
  자동화 근거 예시: 린트 또는 단위 테스트 중 하나만 있고, 커버리지는 수집하지 않습니다.
- 3점: 주요 변경에 대해 실행 가능한 검증 루프가 있습니다.
  자동화 근거 예시: CI에서 빌드와 테스트 또는 린트가 자동 실행되고, 실패 시 머지가 막힙니다.
- 4점: 자동 검증과 수동 시나리오가 적절히 보완됩니다.
  자동화 근거 예시: 테스트 커버리지가 수집되며, 핵심 변경 영역 기준선이 정해져 있습니다.
- 5점: 회귀를 조기에 발견할 가능성이 높고 검증 비용도 낮습니다.
  자동화 근거 예시: CI, 테스트, 커버리지, 린트, 타입 체크가 일관되게 연결되고 품질 게이트로 사용됩니다.

### 6. 문서/계약 명확성

- 0점: 구현과 문서가 거의 연결되지 않습니다.
- 1점: 문서는 있으나 현재 구현과 많이 어긋납니다.
- 2점: 핵심 문서는 있으나 계약 변경 추적이 어렵습니다.
- 3점: 현재 구현 판단에 필요한 문서와 계약이 대체로 존재합니다.
- 4점: 변경 이유, API 계약, 운영 기준이 함께 정리됩니다.
- 5점: 인수인계와 후속 작업이 문서만으로도 상당 부분 가능합니다.

### 7. 운영 준비도

- 0점: 장애 시 확인 경로가 거의 없습니다.
- 1점: 동작은 하지만 로그, 설정, 복구 절차가 불분명합니다.
- 2점: 최소 운영은 가능하나 장애 분석이나 보안 기준이 약합니다.
- 3점: 현재 팀 규모에서 감당 가능한 운영 기준을 충족합니다.
- 4점: 배포, 로그, 오류 처리, 시크릿 관리가 비교적 안정적입니다.
- 5점: 장애 파악과 복구가 빠르고 운영 리스크가 잘 통제됩니다.

## 측정 절차

품질 점수는 아래 순서로 매깁니다.

1. 변경 범위를 확인합니다.
2. 관련 제품 명세, 설계 문서, API/데이터 문서를 함께 확인합니다.
3. 먼저 CI, 테스트, 린트, 커버리지 같은 자동화 지표를 수집합니다.
4. 자동화 지표로 설명되지 않는 부분만 사람 판단으로 보완합니다.
5. 코드와 테스트, 린트, 실행 결과를 기준으로 항목별 점수를 부여합니다.
6. 각 항목마다 한 줄 메모를 남깁니다.
7. 낮은 점수 항목 중 실제 사용자 가치나 운영 리스크가 큰 것을 다음 작업으로 선정합니다.

## 자동화 지표 도입 우선순위

현재 품질 점수표를 더 객관적으로 쓰기 위해 우선 도입을 권장하는 항목은 아래와 같습니다.

1. Backend 테스트 커버리지 수집
2. Frontend 테스트 러너 도입과 최소 화면/유틸 테스트 추가
3. Frontend CI 또는 저장소 루트 CI에서 lint 자동 실행
4. 커버리지 임계치 또는 핵심 패키지 기준선 정의

가정(Assumption): 커버리지 임계치는 도구 도입 직후 높게 잡기보다, 현재 수준을 기준선으로 기록한 뒤 점진적으로 올리는 편이 현실적입니다.

## 점수 해석 규칙

- `0-1점`: 즉시 수정 후보입니다. 기능은 돌아가더라도 다음 작업에서 우선 보완합니다.
- `2점`: 기술 부채 후보입니다. 현재 릴리스는 가능할 수 있지만 이유와 후속 계획을 남깁니다.
- `3점`: 현재 단계에서 수용 가능한 기준선입니다.
- `4-5점`: 당장 리팩토링보다 유지와 재사용을 우선합니다.

다만 평균 점수가 높아도 `제품 적합성`, `테스트/검증 가능성`, `운영 준비도` 중 하나가 `1점 이하`면 리스크가 큰 상태로 봅니다.

## 현재 기준선

현재 저장소의 문서와 구현 상태를 바탕으로 한 거친 기준선은 아래와 같습니다.
이 값은 전체 코드베이스에 대한 정밀 감사 결과가 아니라, 현재 문서 수준에서 파악 가능한 출발점입니다.

| 항목 | 점수 (0-5) | 메모 |
| --- | --- | --- |
| 제품 적합성 | 3 | 제품 목표와 핵심 사용자 흐름 문서는 비교적 명확합니다. |
| 도메인/용어 일관성 | 3 | 핵심 용어 기준은 있으나 구현 전반 정합성 점검은 더 필요합니다. |
| 구조 단순성 | 3 | 소수 팀 운영을 고려한 방향은 명확하지만 세부 런타임 구조는 보강 여지가 있습니다. |
| 변경 용이성 | 2 | 구현 전반의 중복과 결합도는 실제 코드 기준 점검이 더 필요합니다. |
| 테스트/검증 가능성 | 2 | Backend는 CI와 JaCoCo 리포트가 가능하지만 Frontend 자동 테스트와 커버리지는 아직 없습니다. |
| 문서/계약 명확성 | 3 | 상위 기준 문서는 갖춰졌지만 구현 변경과 동기화하는 루틴이 더 필요합니다. |
| 운영 준비도 | 2 | 운영 기준 문서는 있으나 실제 배포/복구 절차의 구체화는 더 필요합니다. |

## 작업 후 기록 템플릿

아래 템플릿을 복사해 기능 작업, 코드 리뷰, 리팩토링 기록에 사용할 수 있습니다.

| 항목 | 점수 (0-5) | 메모 | 후속 액션 |
| --- | --- | --- | --- |
| 제품 적합성 |  |  |  |
| 도메인/용어 일관성 |  |  |  |

| 변경 용이성 |  |  |  |
| 테스트/검증 가능성 |  |  |  |
| 문서/계약 명확성 |  |  |  |
| 운영 준비도 |  |  |  |

가능하면 메모에 아래 자동화 증거를 함께 적습니다.

- 실행한 명령: 예) `./gradlew test`, `npm run lint`
- CI 결과: 통과/실패, 워크플로 이름
- 커버리지: 수집 여부, 비율, 기준선 대비 증감
- 수동 확인: 핵심 시나리오 1-2개

## 리팩토링 우선순위 규칙

- 사용자 흐름을 깨는 항목을 먼저 고칩니다.
- 같은 점수라면 테스트/검증 가능성이 더 낮은 항목을 먼저 고칩니다.
- 구조 개선이 필요해도 운영 복잡도를 크게 높이는 선택은 한 번 더 검토합니다.
- 문서 누락으로 반복 혼선이 생기면 코드 수정과 함께 문서 갱신을 같은 작업으로 묶습니다.
