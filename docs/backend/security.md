# 보안

이 문서는 Matchuri의 인증, 인가, 시크릿 관리 기준을 정리합니다.
현재 목표는 소수 인원이 운영 가능한 단순한 구조를 유지하면서도, 이후 `Spring Security` 기반의 자체 로그인과 `OAuth2` 소셜 로그인을 무리 없이 확장할 수 있도록 기준을 먼저 맞추는 것입니다.

## 문서 목적

- 인증/인가 설계 결정을 구현 전에 팀 기준으로 맞춥니다.
- API 계약, 에러 코드, 회원 모델, 프론트 인증 흐름이 서로 어긋나지 않게 합니다.
- 지금 당장 구현할 항목과 나중에 확정할 항목을 구분합니다.

## 현재 상태

- Backend는 Spring Boot 4 기반입니다.
- 현재 저장소에는 회원/인증 2단계 구현이 진행 중입니다.
- 자체 로그인과 OAuth2 소셜 로그인은 문서 기준으로 2단계 범위에 포함됩니다.
- 공개 API 패턴 초안은 `backend/src/main/resources/application.yaml`에 존재합니다.
- 가정(Assumption): 실제 코드 구현 상태보다 이 문서의 결정이 앞설 수 있으며, 이 경우 내부 실행 계획 기록과 함께 구현에 반영합니다.

## 현재 보안 전제

- Backend는 EC2에 배포합니다.
- DB는 초기에는 EC2 내부 MySQL로 운영합니다.
- Frontend는 Vercel에 배포합니다.
- 시크릿은 Infisical을 기준 저장소로 관리합니다.
- Redis는 현재 즉시 도입하지 않습니다.
- 인증은 세션보다 토큰 기반 구조를 우선합니다.

배포 인프라 기준은 `docs/decisions/backend-deployment-infrastructure.md`, 환경 변수와 운영 절차는 내부 운영 문서를 함께 봅니다.

## 기본 원칙

- 시크릿은 저장소에 커밋하지 않습니다.
- 인증과 인가는 분리해서 생각합니다.
- 입력 검증은 요청 경계에서 수행합니다.
- 실패 응답은 일관되게 주되, 내부 구현 정보는 노출하지 않습니다.
- 구현 복잡도보다 운영 가능성을 우선합니다.
- 회원 식별 모델은 자체 로그인과 소셜 로그인을 함께 수용할 수 있어야 합니다.

## 보호해야 하는 것

### 민감 정보

- DB 접속 정보
- JWT Secret 또는 비대칭 키
- OAuth2 Client ID / Client Secret
- 서버 접속 정보
- 배포 자동화용 시크릿

### 보호 대상 데이터

- 회원 계정 정보
- 로그인 ID, 이메일, 비밀번호 해시
- 회원 취향 및 제한 재료 정보
- 그룹 초대 코드
- 인증 토큰 또는 토큰 메타데이터
- 운영 로그 중 민감 정보가 포함될 수 있는 항목

## 인증 아키텍처 방향

### 목표 구조

- 인증 프레임워크는 `Spring Security`를 사용합니다.
- 자체 로그인은 `loginId + password` 기반으로 처리합니다.
- 소셜 로그인은 `Spring Security OAuth2 Client`를 사용합니다.
- API 인증은 최종적으로 JWT Access Token 기반으로 통일합니다.
- Frontend는 백엔드가 발급한 토큰만 기준으로 로그인 상태를 해석합니다.

### 왜 이 방향을 쓰는가

- 자체 로그인과 소셜 로그인을 각각 임시 구현으로 따로 만들면 이후 통합 비용이 커집니다.
- `Spring Security`를 기반으로 두면 인증 실패 처리, 공개/보호 경로 관리, Principal 주입, OAuth2 콜백 처리를 한 흐름으로 맞출 수 있습니다.
- 운영 인원이 적기 때문에, 커스텀 보안 프레임워크를 직접 만드는 것보다 검증된 표준 구성을 쓰는 편이 안전합니다.

## 회원 식별 모델

### 기준

- `Member`는 서비스에서 최종적으로 사용하는 내부 회원 엔티티입니다.
- 인증 수단이 여러 개여도 서비스 내부 식별자는 하나의 `memberId`를 사용합니다.
- 자체 로그인과 소셜 로그인은 인증 수단이 다를 뿐, 최종 서비스 사용자는 `Member`로 통합합니다.
- 계정 생성과 필수 약관 동의 완료는 같은 개념으로 취급하지 않습니다.

### 권장 모델

- 자체 로그인 계정은 `loginId`, `passwordHash`를 사용합니다.
- 신규 자체 로그인 계정은 회원 생성 전에 검증된 이메일을 요구합니다.
- 소셜 로그인 계정은 `socialProviderType`, 소셜 제공자 사용자 식별자를 `members` 모델 안에 둡니다.
- 현재 2단계 구현에서는 별도 `member_social_accounts` 연결 테이블을 도입하지 않습니다.
- 필수 약관 동의 이력은 `members`가 아니라 별도 `member_agreements` 구조로 관리합니다.

## 필수 약관 동의 정책

### 현재 확정 기준

- 현재는 필수 약관 동의만 고려합니다.
- 필수 약관 종류는 `서비스 이용약관`, `개인정보 처리방침` 두 가지입니다.
- 자체 로그인과 소셜 로그인 신규 가입 모두 같은 필수 약관 동의 기준을 사용합니다.
- 최신 필수 버전은 서버 상수로 관리합니다.
- 약관 동의 이력은 별도 `member_agreements` 테이블로 관리합니다.

### 왜 별도 구조를 쓰는가

- 계정 생성 후 약관 동의가 이어지는 흐름을 자연스럽게 표현할 수 있습니다.
- 회원별로 어떤 약관 타입의 어떤 버전에 동의했는지 남길 수 있습니다.
- 약관 개정 시 재동의 대상 판별이 가능합니다.
- `members` 테이블에 약관 전용 상태 필드를 계속 덧붙이지 않아도 됩니다.

### 서버 판단 기준

- 서버는 회원이 현재 필수 약관 2종의 최신 필수 버전에 모두 동의했는지로 필수 약관 완료 여부를 판단합니다.
- 권장 기준 메서드 예시:
  - `hasCompletedRequiredAgreements(memberId)`
- 로그인, refresh, OAuth2 교환 시 Access Token에는 현재 필수 약관 revision claim을 포함합니다.
- 핵심 API 인가는 요청마다 DB를 다시 조회하지 않고 Access Token claim과 서버의 현재 필수 약관 revision을 비교해 수행합니다.
- 상태 계산은 전체 동의 이력을 정렬 없이 덮어쓰지 않고, 각 필수 약관 타입별 최신 요구 버전 레코드의 존재 여부로 판단합니다.

### 최신 필수 버전 관리

- 최신 필수 버전은 별도 마스터 테이블이 아니라 서버 상수로 관리합니다.
- 예시 방향:
  - `TERMS_OF_SERVICE_REQUIRED_VERSION`
  - `PRIVACY_POLICY_REQUIRED_VERSION`

### 인증/인가에 미치는 영향

- 미동의 회원도 로그인 자체는 허용합니다.
- 다만 필수 약관 2종의 최신 필수 버전에 모두 동의하기 전까지는 핵심 API 접근을 허용하지 않습니다.
- 프론트는 로그인 직후 약관 동의 완료 여부를 확인하고, 미완료면 전용 약관 동의 화면으로 강제 이동합니다.
- 필수 약관 동의 완료 시 서버는 최신 revision을 반영한 새 Access Token을 재발급하며, 프론트는 이를 즉시 교체해야 합니다.

### 계정 병합 원칙

- MVP 단계에서는 자동 계정 병합을 기본 동작으로 두지 않습니다.
- 자체 로그인과 소셜 로그인 간 계정 통합 기능은 고려하지 않습니다.
- 동일 이메일 여부와 무관하게 자동 병합을 하지 않습니다.
- 이유:
- 자체 로그인은 현재 이메일을 회원 식별 기준으로 사용하지 않습니다.
- 추후 자체 로그인에서 이메일 수집 정책이 추가되더라도, 계정 통합 기능은 운영 복잡도와 보안 위험 때문에 도입하지 않습니다.

## 인증 수단별 정책

### 1. 자체 로그인

- 회원 가입 시 `loginId`, `password`를 받습니다.
- 이메일은 현재 자체 로그인 필수 수집 항목이 아닙니다.
- 비밀번호는 평문 저장을 금지하고 해시로만 저장합니다.
- 권장 해시 알고리즘은 `BCrypt`입니다.
- 로그인 실패 응답은 계정 존재 여부를 과도하게 드러내지 않습니다.
- 비활성 회원은 로그인 자체를 차단합니다.
- 자체 로그인으로 생성된 회원도 필수 약관 동의 완료 전에는 핵심 API를 사용할 수 없습니다.

### 2. OAuth2 소셜 로그인

- 현재 지원 소셜 로그인 provider는 `Google`, `Kakao`, `Naver`입니다.
- 추가 provider 확대는 후속 단계에서 검토합니다.
- 최초 소셜 로그인 성공 시:
- 이미 연결된 소셜 계정이 있으면 해당 `Member`로 로그인합니다.
- 연결된 계정이 없으면 신규 회원을 즉시 생성합니다.
- 소셜 로그인 직후 추가 정보 입력은 회원 생성의 선행 조건으로 두지 않고, 로그인 후 온보딩 단계로 분리합니다.
- 소셜 로그인 사용자는 비밀번호를 필수로 갖지 않아도 됩니다.
- 소셜 계정과 자체 계정은 자동 병합하지 않으며, 계정 통합 기능도 현재 범위에서 제공하지 않습니다.
- 신규 소셜 회원도 자체 로그인 회원과 동일하게 필수 약관 동의 완료 전에는 핵심 API를 사용할 수 없습니다.

### 3. 공통 원칙

- 자체 로그인과 소셜 로그인 모두 최종 응답 형식은 가능한 한 동일하게 맞춥니다.
- 로그인 성공 후에는 동일한 Access Token 구조를 발급합니다.
- 인증 수단에 따라 API 인가 규칙이 달라지지 않도록 합니다.
- 로그인 성공과 서비스 이용 가능 상태를 같은 것으로 취급하지 않습니다.

## OAuth2 처리 정책

### 성공 흐름

1. 사용자가 프론트에서 소셜 로그인 버튼을 누릅니다.
2. 프론트는 백엔드 OAuth2 시작 엔드포인트 또는 백엔드가 제공하는 시작 URL로 이동합니다.
3. OAuth2 인증 성공 후 백엔드는 사용자 식별 정보를 확인합니다.
4. 기존 연결 계정이 있으면 해당 회원으로 로그인 처리합니다.
5. 신규 사용자면 즉시 회원을 생성합니다.
6. 백엔드는 프론트 성공 URL로 단기 교환 코드를 포함해 리다이렉트합니다.
7. 프론트는 후속 교환 API를 호출해 Matchuri Access Token을 받습니다.
8. 프론트는 필수 약관 동의 완료 여부를 확인합니다.
9. 미완료면 약관 동의 화면으로 이동합니다.
10. 프론트는 이후 소셜 제공자 토큰이 아니라 Matchuri JWT만 사용합니다.

### 실패 흐름

- 사용자가 제공자 인증을 취소한 경우
- 제공자에서 필수 정보 제공을 거부한 경우
- 지원하지 않는 제공자 또는 잘못된 redirect 요청인 경우
- 이미 차단된 회원 상태인 경우

이 경우 프론트가 처리 가능한 수준의 실패 코드와 메시지를 반환해야 합니다.

### 제공자 토큰 사용 원칙

- 제공자 Access Token은 로그인 처리에 필요한 최소 범위로만 사용합니다.
- Matchuri API 인증에는 제공자 토큰을 직접 사용하지 않습니다.
- 제공자 토큰 저장이 꼭 필요하지 않다면 저장하지 않는 방향을 우선합니다.
- 가정(Assumption): MVP에서는 소셜 로그인 후 외부 제공자 API를 지속 호출할 요구가 크지 않으므로, 장기 저장 필요성이 낮습니다.

### 토큰 전달 원칙

- OAuth2 로그인 성공 후 Access Token을 URL에 직접 노출하지 않습니다.
- 프론트 복귀 시에는 짧게 유효한 단기 교환 코드를 사용합니다.
- Refresh Token은 `HttpOnly Secure Cookie` 기준 처리를 우선합니다.

## JWT 정책

### 기본 방향

- Access Token 중심으로 시작합니다.
- 현재 구현은 Access Token + Refresh Token 구조를 사용합니다.
- Redis를 아직 쓰지 않으므로 토큰 무효화 정책은 단순하게 시작합니다.
- Redis 기반 토큰 무효화는 현재 단계에서 도입하지 않습니다.
- JWT 서명은 현재 단일 백엔드 운영 구조를 기준으로 대칭키 방식을 사용합니다.

### Access Token

- 용도: API 인증
- 포함 정보: `memberId`, `role`, `loginId`, 필수 약관 revision 등 최소 claim
- 만료 시간: 짧게 유지합니다.
- 권장 시작점: 30분 ~ 2시간 범위에서 확정

### Refresh Token

- 현재 구현은 `로그인 단위 다중 Refresh Token 저장`을 사용합니다.
- 공통 원칙:
- 자체 로그인과 소셜 로그인은 같은 Refresh Token 정책을 사용합니다.
- 로그아웃은 Access Token 즉시 전역 무효화가 아니라 Refresh Token 차단 중심으로 정의합니다.
- 브라우저 세션 복구를 위해 `POST /api/v1/auth/refresh` 재발급 API를 제공합니다.
- 모바일과 웹 동시 사용 가능성을 고려해, 새 로그인 시 기존 다른 기기의 Refresh Token을 일괄 무효화하지 않습니다.
- 현재 구현은 재발급 시 기존 Refresh Token을 같은 로그인 세션 안에서 회전합니다.
- Refresh Token 회전의 추가 보안 정책과 세션 관리 UI는 후속 단계에서 구체화합니다.

### 로그아웃

- Redis 미도입 단계에서는 즉시 전역 무효화보다 단순 정책을 우선합니다.
- Access Token은 짧은 만료 시간으로 관리합니다.
- Refresh Token을 운영한다면 서버 저장소에서 삭제 또는 폐기 상태로 전환합니다.
- 로그아웃의 의미는 "이후 재발급 차단"을 기준으로 정의합니다.
- 현재 단계에서 로그아웃은 `현재 로그인 세션의 Refresh Token 삭제 + Refresh Token 쿠키 제거`를 의미합니다.
- 이미 발급된 Access Token은 블랙리스트 처리하지 않으며, 만료 시점까지는 보호 API 호출에 사용할 수 있습니다.

### 다중 기기 로그인

- 다중 로그인은 허용합니다.
- 현재 단계에서는 로그인 단위 다중 Refresh Token 저장을 전제로 합니다.
- 기기별 세션 식별이나 동시 접속 수 제한은 아직 두지 않습니다.
- 재발급 API와 세션 관리 UI는 후속 단계에서 구체화합니다.

### 강제 로그아웃

- MVP 단계에서는 강제 로그아웃 기능을 제공하지 않습니다.
- 관리자 수동 종료, 비밀번호 변경 시 전역 만료, 의심 세션 차단 같은 정책은 후속 단계에서 검토합니다.

### JWT 키 전략

- 현재는 HMAC 기반 대칭키 서명을 사용합니다.
- 비대칭키 전환은 다중 서비스 검증 또는 운영 키 배포 요구가 생길 때 검토합니다.

## 인가 정책

### 공개 API

- 회원 가입
- 로그인
- 로그인 ID 중복 확인
- 이메일 인증 코드 발송/확인
- 로그인 ID 찾기와 비밀번호 재설정
- OAuth2 시작 및 콜백 처리 엔드포인트
- 공개 문서 엔드포인트
- 헬스 체크 엔드포인트

### 보호 API

- 약관 상태 조회/동의 제출을 제외한 핵심 API는 필수 약관 동의 완료를 전제로 합니다.
- `members/me`
- 취향 프로필 관리
- 개인 추천 요청/조회/선택
- 그룹 생성/참여/탈퇴
- 그룹 추천 세션 시작
- 그룹 투표 및 최종 확정

### 인가 원칙

- 본인 정보는 본인만 접근 가능해야 합니다.
- 필수 약관 미동의 회원은 로그인되어 있더라도 핵심 API에서 차단해야 합니다.
- 그룹 관련 기능은 그룹 멤버 여부를 반드시 확인해야 합니다.
- 방장 전용 동작은 일반 멤버와 구분합니다.
- 어드민 API는 서비스 API와 경로와 문서 수준에서 분리합니다.

## Spring Security 구성 원칙

### 필수 구성 요소

- `SecurityFilterChain`
- 비밀번호 인코더
- JWT 인증 필터 또는 인증 컨버터
- 인증 실패 핸들러
- 인가 실패 핸들러
- OAuth2 로그인 성공/실패 핸들러

### 경로 관리

- 공개 경로는 화이트리스트 방식으로 최소한만 엽니다.
- 나머지 서비스 API는 기본적으로 인증 필요 상태를 기본값으로 둡니다.
- 현재 `application.yaml`의 `public-api-patterns`는 구현 시작점으로 쓰되, 실제 Security 설정이 생기면 한 곳에서 일관되게 관리합니다.
- 인증 이후에는 "로그인 필요"와 "필수 약관 동의 완료 필요"를 별도 조건으로 구분합니다.

### API 문서 엔드포인트 보호 기준

- Swagger UI 경로는 `/docs/swagger-ui.html`입니다.
- OpenAPI 문서 경로는 `/docs/openapi`입니다.
- 현재 애플리케이션은 `/docs/**`를 공개 경로로 둡니다.
- `local`에서는 그대로 익명 접근을 사용합니다.
- `dev`와 `prod`의 외부 노출 통제는 애플리케이션 권한이 아니라 Nginx Basic Auth가 담당합니다.
- 가정(Assumption): staging과 prod는 모두 Nginx를 외부 진입점으로 사용합니다.
- 운영 기준 절차는 내부 운영 문서를 따릅니다.

### 세션 정책

- API 서버는 상태 없는 인증 흐름을 우선합니다.
- 현재 보안 필터 체인은 `SessionCreationPolicy.STATELESS` 기반 구성을 사용합니다.
- 다만 OAuth2 시작~callback 사이의 임시 authorization request 상태는 예외적으로 서버 세션에 저장합니다.
- 이는 서비스 로그인 전체를 세션 기반으로 바꾼 것이 아니라, OAuth2 handshake 검증에 필요한 짧은 중간 상태만 서버가 보관하는 구조입니다.
- 현재 단일 EC2 운영에서는 관리 가능한 선택이며, 추후 다중 인스턴스 확장 시에는 세션 공유 전략을 다시 검토합니다.

## 비밀번호 정책

- 비밀번호는 해시로만 저장합니다.
- 권장 구현은 `BCryptPasswordEncoder`입니다.
- 평문 비밀번호는 로그에 남기지 않습니다.
- 최소 길이와 허용 문자 정책은 회원 가입 API 문서에 명시합니다.
- 비밀번호 비교 실패 시 상세 원인은 노출하지 않습니다.

## 이메일 인증과 계정 복구 정책

로그인 ID 찾기와 비밀번호 재설정은 이메일 인증을 통과한 경우에만 허용합니다.
상세 설계와 API 계약은 `docs/decisions/email-verification-and-account-recovery.md`, `docs/api/auth-email-verification.md`를 기준으로 봅니다.

현재 결정 기준:

- 신규 자체 로그인 회원가입은 이메일 인증을 회원 생성 전 조건으로 둡니다.
- MVP 프로토타입에서는 회원가입 흐름 단순화를 위해 `SIGNUP` 인증 코드 발송 요청에서 이미 가입된 자체 로그인 이메일이면 중복 이메일 오류를 명시적으로 반환합니다.
- 로그인 ID 찾기와 비밀번호 재설정 목적의 인증 코드 발송 요청은 계정 존재 여부를 응답으로 드러내지 않습니다.
- 인증 코드는 평문 저장하지 않습니다.
- 계정 복구용 verification token은 JWT가 아니라 DB 저장형 랜덤 토큰으로 발급합니다.
- verification token 원문은 응답 시 한 번만 노출하고, 서버에는 해시와 만료/사용 상태만 저장합니다.
- 인증 코드와 계정 복구용 verification token 원문은 로그에 남기지 않습니다.
- 비밀번호 재설정 성공 후 자동 로그인하지 않습니다.
- 한 이메일에 여러 자체 로그인 ID를 허용하지 않습니다.
- 비밀번호 재설정 성공 시 해당 회원의 기존 refresh token 전체를 폐기합니다.

## 예외 및 에러 응답 정책

### 구분 원칙

- 인증 실패는 `401`
- 인가 실패는 `403`
- 리소스 없음은 `404`
- 중복/상태 충돌은 `409`

### 인증 관련 대표 코드

- `AUTH_TOKEN_MISSING`
- `AUTH_TOKEN_INVALID`
- `AUTH_TOKEN_EXPIRED`
- `AUTH_REFRESH_TOKEN_MISSING`
- `AUTH_REFRESH_TOKEN_INVALID`
- `AUTH_REFRESH_TOKEN_EXPIRED`
- `AUTH_LOGIN_FAILED`
- `AUTH_FORBIDDEN`
- `AUTH_LOGOUT_FAILED`

### 응답 원칙

- 에러 응답에는 내부 클래스명, SQL, 스택트레이스를 노출하지 않습니다.
- 로그인 실패 메시지는 `loginId`와 비밀번호 중 무엇이 틀렸는지 구분하지 않습니다.
- OAuth2 실패도 프론트에서 처리 가능한 범위의 코드와 메시지로 정리합니다.

## 시크릿 관리

### 관리 기준

- 프로젝트에서 사용하는 모든 시크릿은 Infisical을 단일 관리 기준으로 사용합니다.
- 환경은 `development`, `staging`, `production`으로 분리합니다.
- Infisical은 시크릿의 source of truth 역할을 합니다.

### 저장 원칙

- `.env` 파일은 저장소에 커밋하지 않습니다.
- 백엔드 CD의 애플리케이션 시크릿은 GitHub Actions OIDC로 Infisical에서 실행 시점에 가져옵니다.
- EC2 운영 환경에서는 서버 환경 변수 또는 안전한 설정 파일로 분리합니다.
- OAuth2 Client Secret, JWT Secret, DB 비밀번호는 별도 키로 관리합니다.

### 로컬 개발 방식

- `frontend`, `backend` 모두 로컬 실행 전에 Infisical CLI를 통해 최신 시크릿을 동기화합니다.
- 로컬 `.env`는 개발 편의를 위한 산출물이며 저장소에 포함하지 않습니다.

### 배포 및 런타임 주입

- Backend CD는 `Infisical Secrets Action + GitHub Actions OIDC`를 사용합니다.
- GitHub Actions workflow는 `id-token: write` 권한으로 GitHub OIDC 토큰을 받고, Infisical identity와 교환해 단기 액세스 권한을 획득합니다.
- AWS 리소스 접근이 필요한 배포 단계는 `GitHub Actions OIDC -> AWS IAM Role Assume`을 기본으로 사용합니다.
- 배포 job은 가져온 시크릿으로 `/etc/matchuri/backend.env`를 렌더링하거나 갱신한 뒤 EC2 배포 스크립트를 실행합니다.
- 애플리케이션 시크릿을 GitHub repository secret에 장기 저장하지 않습니다.
- 배포 설정값은 GitHub Environment variables로 두고, 애플리케이션 시크릿과 Infisical 연결 정보만 GitHub Environment secrets로 둡니다.
- SSH `22` 전체 공개는 임시 배포 검증용으로만 허용하고, 장기 운영 자동 배포는 AWS SSM Run Command / Session Manager 전환을 기본안으로 사용합니다.
- Frontend 배포는 Vercel 기본 Git 배포를 사용합니다.
- 프론트 공개 환경 변수는 Infisical `App Connection + Secret Sync`를 통해 Vercel 프로젝트 환경 변수로 동기화합니다.
- 프론트 환경 변수의 source of truth는 Infisical이며, Vercel 콘솔의 수동 입력은 예외 상황으로만 사용합니다.

## 로그와 감사 기준

- 비밀번호, Access Token, Refresh Token 원문은 로그에 남기지 않습니다.
- OAuth2 제공자 응답 원문도 민감 정보 마스킹 없이 저장하지 않습니다.
- 인증 실패, 권한 실패, 토큰 검증 실패는 원인 추적이 가능하도록 구조화 로그를 남깁니다.
- 인증 관련 운영 로그에는 필요 시 요청 IP를 포함합니다.
- 권장 최소 로그 필드:
- 이벤트 종류 (`login_success`, `login_failed`, `token_invalid`, `token_expired`, `logout`)
- `memberId` 또는 식별 가능한 최소 회원 정보
- 인증 수단 (`local`, 소셜 제공자 유형)
- 실패 코드
- 요청 시각
- 요청 IP
- 운영 환경 변경과 수동 데이터 수정은 가능한 한 기록이 남는 방식으로 수행합니다.

## API 문서화 기준

- 인증 필요 여부를 각 API 명세에 명시합니다.
- `Authorization: Bearer` 요구 여부를 OpenAPI에 반영합니다.
- OAuth2 로그인은 프론트 리다이렉트 흐름과 백엔드 콜백 흐름을 함께 문서화합니다.

## 체크리스트

- 시크릿이 저장소에 커밋되지 않는다.
- 비밀번호는 해시로만 저장된다.
- 자체 로그인과 소셜 로그인이 하나의 회원 식별 모델로 수렴한다.
- 필수 약관 동의 완료 여부가 `member_agreements` 기준으로 일관되게 판단된다.
- 인증과 권한 검사가 `Spring Security` 경계에서 일관되게 수행된다.
- 공개 API와 보호 API가 분리되어 있다.
- 에러 응답에 내부 구현 정보가 노출되지 않는다.
- 인증 관련 에러 코드와 API 문서가 일치한다.
- 운영 환경 변수 관리 방식이 정리되어 있다.
