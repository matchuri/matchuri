# 신뢰성

이 문서는 Matchuri를 운영할 때 장애를 어떻게 줄이고, 문제가 생기면 어떻게 확인하고, 어떻게 복구할지를 정리합니다.
현재는 대규모 서비스 기준이 아니라, 2인 팀이 실제로 감당 가능한 최소 신뢰성 기준을 먼저 정의합니다.

## 현재 운영 전제

- Backend는 EC2 단일 서버에 배포합니다.
- DB는 초기에는 EC2 내부 MySQL로 운영합니다.
- Frontend는 Vercel에 배포합니다.
- Redis는 즉시 도입하지 않고, 추후 성능 개선이 필요해질 때 도입합니다.

배포 구조의 근거는 `docs/decisions/backend-deployment-infrastructure.md`, 실제 절차는 내부 운영 문서를 기준으로 봅니다.

이 전제는 운영 복잡도를 낮추는 대신 단일 장애 지점이 존재한다는 뜻입니다.

## 신뢰성 목표

현재 단계의 목표는 "무중단 고가용성"이 아니라 "장애를 빨리 파악하고, 단순한 절차로 복구 가능한 상태"를 만드는 것입니다.

## 장애 모델

현재 구조에서 먼저 대비해야 하는 장애는 아래와 같습니다.

### 1. 애플리케이션 기동 실패

- 배포 후 Spring Boot 애플리케이션이 정상 실행되지 않는 경우
- 잘못된 환경 변수, 포트 충돌, DB 연결 실패가 원인일 수 있음

### 2. DB 연결 실패

- MySQL이 죽었거나 연결 정보가 잘못된 경우
- EC2 내부에서 API와 DB가 함께 운영되므로 영향 범위가 큼

### 3. 배포 실패

- GitHub Actions는 성공했지만 EC2 반영이 실패하는 경우
- 신규 버전 실행은 되었지만 정상 기동 확인이 안 되는 경우

### 4. 서버 자원 부족

- EC2 자원 부족으로 API 응답이 느려지거나 DB까지 함께 영향을 받는 경우

### 5. 외부 연동 또는 잘못된 입력으로 인한 요청 실패

- 잘못된 요청 값, 인증 오류, 비즈니스 예외가 반복적으로 발생하는 경우

## 기본 원칙

- 장애를 완전히 막는 것보다, 원인을 빨리 좁힐 수 있는 구조를 우선합니다.
- 복잡한 자동 복구보다, 사람이 이해 가능한 단순 복구 절차를 먼저 만듭니다.
- 배포와 운영 절차는 문서로 남기고 반복 가능한 방식으로 유지합니다.
- 새 기술은 신뢰성을 높이는 효과가 분명할 때 도입합니다.

## 최소 운영 기준

### 배포 기준

- 배포는 가능한 한 GitHub Actions 기반 자동화로 수행합니다.
- 배포 후 애플리케이션 기동 확인 절차가 있어야 합니다.
- 실패 시 이전 버전을 다시 실행할 수 있는 단순 복구 경로를 유지합니다.

### 실행 기준

- 애플리케이션은 서버 재시작 후 다시 올릴 수 있는 방식으로 운영합니다.
- 환경 변수는 코드에 하드코딩하지 않고 서버 또는 CI 비밀값으로 관리합니다.
- 로그를 확인할 수 있는 위치와 명령을 정리합니다.

### 데이터 기준

- DB 백업은 최소 기준이라도 반드시 필요합니다.
- 스키마 변경은 문서와 함께 관리합니다.
- 개발 편의를 위해 운영 DB를 쉽게 덮어쓰는 방식은 피합니다.

## 로그와 관측

초기 단계에서는 복잡한 모니터링 시스템보다 아래 수준을 먼저 확보합니다.

### 반드시 가능한 것

- 애플리케이션 시작 성공/실패 여부 확인
- 최근 에러 로그 확인
- 배포 성공/실패 여부 확인
- 서버 상태 확인
- DB 연결 실패 여부 확인

### 나중에 도입할 수 있는 것

- 에러 알림 자동화
- 메트릭 수집
- 응답 시간 추적
- 외부 모니터링 도구 연동

## 오류 처리 원칙

- 잘못된 요청은 4xx로 명확히 구분합니다.
- 서버 내부 문제는 5xx로 응답합니다.
- 에러 코드는 도메인 기준으로 일관되게 관리합니다.
- 예상 가능한 실패는 메시지를 통일하고, 로그에는 원인 추적 정보가 남아야 합니다.

## 타임아웃과 재시도

초기에는 무분별한 재시도를 넣기보다, 실패를 명확히 드러내는 편이 낫습니다.

### 현재 원칙

- DB나 외부 연동 호출은 무한 대기를 피합니다.
- 자동 재시도는 꼭 필요한 지점에만 제한적으로 둡니다.
- 재시도보다 먼저 "왜 실패했는지 추적 가능한 로그"를 확보합니다.

## 백업과 복구

현재 구조에서 가장 중요한 복구 대상은 MySQL 데이터입니다.

### 최소 복구 원칙

- DB 백업 주기와 보관 위치를 정합니다.
- 배포 실패 시 이전 애플리케이션 버전을 다시 올릴 수 있어야 합니다.
- 운영 서버 접근과 복구 절차는 문서화합니다.

### 나중에 보강할 수 있는 것

- 자동 백업 검증
- 스냅샷 기반 복구
- DB 분리 운영

## Redis 관련 방침

Redis는 지금 당장 신뢰성 필수 구성요소가 아닙니다.
현재는 Redis 없이도 동작 가능한 구조를 먼저 만들고, 아래 상황이 분명해질 때 도입을 검토합니다.

- 실시간 상태 동기화 요구가 커질 때
- 캐싱 없이 응답 시간이 문제될 때
- 세션/토큰 관리 복잡도가 증가할 때

즉, 현재 신뢰성 전략은 "Redis가 없어서 불안한 구조"가 아니라, "Redis 없이도 단순하게 운영 가능한 구조"를 먼저 만드는 것입니다.

## 체크리스트

- 배포 후 정상 기동 확인 절차가 있다.
- 로그 확인 위치를 알고 있다.
- DB 백업 방식이 정리되어 있다.
- 운영 환경 변수 관리 방식이 정리되어 있다.
- 배포 실패 시 되돌리는 절차가 있다.
- 단일 서버 구조의 한계를 팀이 인지하고 있다.

## 현재 단계의 현실적인 목표

현재 Matchuri에서 중요한 것은 완벽한 고가용성이 아니라 아래 세 가지입니다.

- 장애 원인을 빨리 확인할 수 있을 것
- 배포 실패 시 당황하지 않고 복구할 수 있을 것
- 소수 인원이 유지 가능한 수준으로 운영 복잡도를 억제할 것
