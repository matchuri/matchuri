# 데이터 정의서 템플릿

이 템플릿은 `docs/data/` 아래 현재 구현 기준 테이블 정의서를 작성할 때 사용합니다.
구현 전 설계 초안에도 사용할 수 있지만, 문서 상태를 반드시 명확히 표시합니다.
사람과 에이전트가 모두 같은 기준으로 구현, 검토, 문서 갱신을 진행할 수 있도록 기준 소스와 확인 순서를 함께 적습니다.

## 작성 원칙

- 현재 구현 기준 문서와 구현 전 초안을 한 문서 안에서 섞지 않습니다.
- 구현과 문서가 다르면 먼저 구현을 확인하고, 문서가 늦었는지 구현이 누락됐는지 구분합니다.
- 오래 유지될 사실만 적고, 일회성 작업 맥락은 실행 계획 문서에 둡니다.
- 미확정 내용은 `가정(Assumption):`으로 분리합니다.
- 테이블/컬럼/enum/상태값 이름은 코드와 같은 표기를 사용합니다.

## 문서 상태

- 상태: 현재 구현 기준 / 구현 전 초안 / 폐기 예정 참고 문서
- 기준일: YYYY-MM-DD
- 담당 영역: member / auth / menu / image / recommendation / group / 기타
- 기준 소스:
  - JPA Entity:
  - DDL / migration / init SQL:
  - 관련 seed:
  - 관련 API 문서:

## 문서 목적

- 이 문서가 설명하는 데이터 영역을 적습니다.
- 코드 구현자가 이 문서를 보고 확인해야 할 핵심 판단을 적습니다.
- 다른 문서와 책임이 겹치면 이 문서가 맡는 범위를 적습니다.

## 현재 확정 전제

- 현재 구현 또는 합의된 운영 전제를 적습니다.
- 구현과 다른 목표/가정은 이 섹션에 섞지 않습니다.
- 미확정 내용은 `가정(Assumption):`으로 별도 표시합니다.
- MVP 범위를 넘는 확장은 현재 정의처럼 쓰지 않습니다.

## `table_name`

### 역할

- 이 테이블이 저장하는 데이터와 책임을 적습니다.
- 다른 테이블에 저장하지 않는 이유가 있으면 짧게 적습니다.

### 컬럼 정의

| 컬럼명 | 타입 | NULL | 기본값 | 제약/기타 | 설명 |
| --- | --- | --- | --- | --- | --- |
| `id` | bigint | N |  | PK, auto increment | 식별자 |
| `created_at` | datetime | N |  | auditing | 생성 일시 |
| `updated_at` | datetime | N |  | auditing | 수정 일시 |

컬럼 작성 기준:

- Java 필드명보다 DB 컬럼명을 우선 적습니다.
- nullable, length, default, enum 저장 방식은 코드 또는 DDL과 대조합니다.
- `created_at`, `updated_at`이 없는 로그성 테이블이면 이유를 적습니다.
- JSON 컬럼은 저장 목적, 대표 key, 변경 가능성을 간단히 적습니다.

### 운영 기준

- 중복 호출, 멱등성, 상태 충돌, 만료 처리, lazy expire 여부를 적습니다.
- 물리 삭제 대신 상태 전환을 쓰는지 적습니다.
- seed, 관리자 수정, 배치 정리 기준이 있으면 적습니다.
- 상태 변경 API가 관련되면 권한 없음, 이미 처리됨, 만료됨 상황을 적습니다.

### 현재 제외 범위

- 지금 구현하지 않는 컬럼, 테이블, 제약, 배치, 운영 UI를 적습니다.
- 나중에 검토할 내용은 현재 정의처럼 쓰지 않습니다.

## 영역 공통 판단

- 여러 테이블에 공통으로 적용되는 판단을 적습니다.
- 예: 삭제보다 비활성화 우선, 회원당 1개만 유지, 최신 row 기준 사용 등.

## 코드 변경 시 확인할 것

- 엔티티 필드, 컬럼 표, DDL/init SQL이 일치하는지 확인합니다.
- FK, unique, nullable, default, length, enum 저장 방식이 일치하는지 확인합니다.
- Repository query와 service write path가 문서의 주요 쿼리/쓰기 패턴과 어긋나지 않는지 확인합니다.
- enum 값 변경 시 API 문서, OpenAPI 메타데이터, 프론트 계약 영향이 있는지 확인합니다.
- 상태 변경 API는 중복 호출, 권한 없음, 상태 충돌, 만료 처리를 확인합니다.
- seed 기준 데이터가 바뀌면 seed SQL과 문서의 운영 기준을 함께 확인합니다.
- 데이터 모델 변경 시 `docs/data/index.md`, `docs/data/implemented-jpa-data-model.md` 갱신 여부를 확인합니다.

## 에이전트 작업 체크리스트

- 작업 시작 전 이 문서의 `문서 상태`와 `기준 소스`를 먼저 확인합니다.
- 현재 구현 기준 문서라면 코드와 DDL을 먼저 대조하고 문서를 갱신합니다.
- 구현 전 초안이라면 현재 구현처럼 단정하지 않고 `권장`, `가정(Assumption):`을 사용합니다.
- 중복 문서가 보이면 바로 내용을 합치기보다 각 문서의 책임을 먼저 구분합니다.
- `docs/data/implemented-jpa-data-model.md`는 빠른 요약으로 보고, 컬럼 단위 근거는 개별 `docs/data/*.md`와 코드를 우선합니다.
- 링크를 추가하거나 바꾸면 상대 경로가 맞는지 확인합니다.
